================================================================================
作成日 : 2026-07-20
前提   : Next.js + Lark Base + Anthropic API（@anthropic-ai/sdk 0.71）の
         既存構成を活かす。社内AIチャット(#32)で確定した
         「セキュリティ最優先」の方針と整合させる。

1. 現状 ― すでにAIは3か所で動いているが、いずれも「単発呼び出し」
================================================================================
  lib/ai-models.ts
    OCR_SCHEDULE  : claude-sonnet-4-6 （工程表PDFの読取。thinking使用）
    TEXT_ANALYSIS : claude-haiku-4-5  （売上/原価/経費の総評生成）
    ＋ createMessageWithFallback() でモデル廃止時に自動フォールバック
    ＋ /api/health/ai-models で廃止を事前検知

  使用箇所
    /api/ai-analysis      売上BI・赤字分析・コピー経費の総評   max_tokens 1024
    /api/ai-cost-analysis 製番別の原価コメント                 max_tokens 1024
    /api/ocr/schedule     工程表PDF → 社内工程表フィールド更新 max_tokens 16000

■ 評価
  モデル廃止に強い設計（チェーン＋ヘルスチェック）は良い土台。
  ただし3つとも「1回投げて1回返す」だけで、エージェント（＝モデルが自分で
  ツールを選び、複数手順を回す）ではない。
  逆に言えば、この3つは今のままで正しい。全部をエージェント化する必要はない。

■ エージェントにすべきか の判断基準
  次の4つが全部YESのときだけエージェント化する。
    複雑さ   : 手順を事前に全部書き下せないか？
    価値     : コストと待ち時間に見合うか？
    実現性   : Claudeがその作業を実際にこなせるか？
    誤りコスト: 間違えても検知・巻き戻しができるか？
  1つでもNOなら、単発呼び出しか、コードで手順を固定したワークフローで足りる。

2. 提案するエージェント（優先度順）
================================================================================

--------------------------------------------------------------------------------
【1】参考図・仕様提案エージェント（建築系）        優先度: ★★★  難易度: 中
--------------------------------------------------------------------------------
■ 何をするか
  営業が客先で聞いた条件（用途・間口・桁行・軒高・所在地・予算感）を自然文で
  投げると、エージェントが自分で調べて提案一式を返す。
    1. 所在地から基準風速・垂直積雪量を引く（既存 /api/eigyo/kijun-fusoku）
    2. その条件に合う参考図面を台帳4,268件から検索（既存 /api/eigyo/sankou-zu）
    3. 類似の過去案件と受注金額・原価を引く（既存 /api/baiyaku, /api/cost-analysis）
    4. 候補3案を根拠つきで提示（図面番号・過去製番・概算レンジ）

■ なぜ最優先か
  ・データが全部すでにMembrySubSystem内にある（新規整備がほぼ不要）
  ・「4タブ×多数の条件を手で埋める」現UIの負担が、自然文1行に変わる
  ・営業の属人性（ベテランしか過去図面を思い出せない）が解消する
  ・4つの判断基準を全部満たす。誤りコストも低い（提案であって発注ではない）

■ ツール定義（既存APIをそのまま道具にする）
    search_sankou_zu   参考図台帳検索。建屋条件で絞り込む
    get_kijun_fusoku   都道府県・市区町村から基準風速と積雪量を取得
    search_baiyaku     過去の売約を条件検索
    get_cost_analysis  製番の原価実績を取得
  ※ 各ツールのdescriptionには「何をするか」だけでなく
    「いつ呼ぶか」を明記する。最近のモデルはツールを控えめに使うため、
    トリガー条件を書くと呼び出し率が目に見えて上がる。

■ 重要な設計判断: ベクトルDB（RAG）は作らない
  参考図台帳は構造化データ（列が定義されたLarkテーブル）。
  ベクトル検索より、既存の検索APIをツールとして渡すほうが
  正確・安価・保守が楽。RAGを検討するのは「非構造の長文」だけでよい。

--------------------------------------------------------------------------------
【2】現場日報 要約・リスク検知エージェント（建築系）優先度: ★★★  難易度: 中
--------------------------------------------------------------------------------
■ 何をするか
  外注業者から上がる日報（テキスト＋現場写真）を毎日まとめて読み、
    ・案件ごとの進捗サマリ（工務課の朝会用）
    ・安全上の懸念（保護具未着用、足場、開口部、高所）
    ・品質上の懸念（施工手順の逸脱、養生不足）
    ・工程遅延の兆候（同じ工種が想定より続いている）
  を抽出してLarkへ通知する。

■ なぜ効くか
  ・日報は毎日増えるが、全部を人が写真まで見るのは現実的に不可能
  ・「見落とし」がそのまま労災・手直しになる領域
  ・機能ギャップ分析で挙げた安全衛生パッケージの実質的な中身になる

■ 設計上の要点
  ・毎日の定期実行。夜間バッチが自然（既存の /api/batch/* と同じ枠組み）
  ・出力は必ず構造化する（structured outputs / json_schema）。
    「案件番号・懸念区分・根拠となる写真ID・重大度」の形でLarkへ書き戻す
  ・AIの指摘は「一次スクリーニング」であり判断ではない、と明示する。
    自動で是正指示を出さず、人が確認してから【提案3】の是正管理に載せる

■ 画像の解像度が効く領域
  現行のOCRはSonnet 4.6（画像の長辺1568px上限）。
  Opus 4.8 は長辺2576pxまで扱え、座標も実ピクセルと1:1で対応する。
  現場写真・図面のような「細部が意味を持つ画像」では差が出るため、
  この用途はOpus 4.8を推奨する（詳細は第4章）。

--------------------------------------------------------------------------------
【3】品質ナレッジ・エージェント（製造系）          優先度: ★★☆  難易度: 低
--------------------------------------------------------------------------------
■ 何をするか
  「膜の端部から雨が入る」といった相談に対し、過去の
  不具合情報・顧客要求事項変更届・設計依頼を横断検索し、
  類似事例と当時の対処・再発防止策を根拠つきで返す。

■ なぜ効くか
  ・データはすでにある（QUALITY_ISSUES / CUSTOMER_REQUESTS）
  ・不具合情報は「登録されるが読まれない」典型。検索性が価値になる
  ・難易度が低く、エージェント運用の練習台として最適

■ 位置づけ
  #32で確定した社内AIチャット（Lark Bot）に、この検索ツールを足す形が自然。
  新しいUIを作らず、既存の会話導線に機能を1つ増やすだけで済む。

--------------------------------------------------------------------------------
【4】KPI異常検知・PDCA支援エージェント（製造系）   優先度: ★★☆  難易度: 中
--------------------------------------------------------------------------------
■ 何をするか
  月次で生産本部KPIを読み、
    ・目標未達（赤・黄）のKPIを特定
    ・過去実績・関連KPI・進行中の施策を自分で引いて原因仮説を立てる
    ・翌月アクションの草案を作る（PDCAの「A」の下書き）
  月次会議の資料を、人が確認・修正する前提で先に用意する。

■ 現行との違い
  今の /api/ai-analysis は「渡された数字に総評を書く」だけ。
  エージェント化すると「気になる数字を自分で深掘りする」ようになる。

■ 注意
  KPIの判定ロジック（lib/kpi の aggregate / judge）は決定論的な純関数であり、
  ここをAIに置き換えてはいけない。AIの仕事は「判定結果の解釈と仮説」であって、
  計算ではない。数値計算は必ず既存関数の結果をツール経由で渡す。

--------------------------------------------------------------------------------
【5】書類ドラフト・エージェント（共通）            優先度: ★★☆  難易度: 低〜中
--------------------------------------------------------------------------------
■ 何をするか
  ・工事写真台帳の情報欄（工種・撮影内容）を日報の作業内容から自動生成
    → 現在は日報値をそのまま初期値にしているが、台帳向けの表現に整えられる
  ・施工計画書・是正報告書・KY活動記録の下書き生成
  ・引渡し書類一式の不足チェック

■ なぜ低リスクか
  出力が必ず人のレビューを経る。誤りコストが小さく、失敗しても巻き戻せる。

--------------------------------------------------------------------------------
【6】業務オペレーター・エージェント（共通）        優先度: ★☆☆  難易度: 高
--------------------------------------------------------------------------------
■ 何をするか
  「A社の受注残を出して」「この製番の工程を1週間後ろにずらして」といった
  自然文の指示を、MembrySubSystemのAPIを叩いて実行する。

■ 最も「エージェントらしい」が、最後に回すべき理由
  ・書き込み系ツールを持たせた瞬間、リスクの質が変わる
  ・プロンプトインジェクション（日報や書類の本文に仕込まれた指示）で
    意図しない更新が走りうる
  ・当面は「読み取り専用ツールのみ」で始め、書き込みは
    人が承認してから実行する形に限定するのが現実的

■ もし作るなら
  ・書き込み系ツールは必ず承認ゲートを通す（Tool Runnerの承認フックで実装可）
  ・lib/menu-access.ts の requireMenuAccess() をツール実行前に必ず通し、
    「そのユーザーが画面で操作できないことは、AIにも操作させない」を徹底する

3. アーキテクチャ ― どこでエージェントを動かすか
================================================================================
選択肢は3つ。それぞれ「ループを誰が書くか」「どこで動くか」が違う。

┌────────────────────┬──────────────┬──────────────┬────────────────────────┐
│ 方式               │ ループ       │ 実行場所     │ 向く用途               │
├────────────────────┼──────────────┼──────────────┼────────────────────────┤
│ 単発API呼び出し    │ 不要         │ 自社(Amplify)│ 総評・要約・分類・OCR  │
│ Tool Runner        │ SDKが提供    │ 自社(Amplify)│ 提案1〜5。★推奨★      │
│ Managed Agents     │ Anthropic    │ Anthropic側  │ 長時間の自律実行       │
└────────────────────┴──────────────┴──────────────┴────────────────────────┘

■ 推奨: Tool Runner（client.beta.messages.tool_runner）
  理由:
  (1) セキュリティ方針と整合する
      Managed Agents はAnthropic側のコンテナでツールが動く。
      #32で「セキュリティ最優先」を理由に埋め込みWeb版を撤去し、
      既存Lark Botへの導線(方式C)に絞った判断と、同じ基準で考えるべき。
      社内の売約・原価・人事データを扱うエージェントは自社ホストが筋。
  (2) 実装が軽い
      ツール関数を書くだけでよく、ループを自分で書かなくてよい。
      承認ゲート・ログ・結果の書き換え・リトライは
      1ターンごとのフックで差し込めるため「制御が効かない」ことはない。
  (3) 既存資産がそのまま使える
      app/api/* の各エンドポイントを薄くラップすればツールになる。
      権限は menu-access.ts が既にサーバ側で強制している。

■ 各エージェントの置き場所
    提案1 参考図提案    → Lark Bot から呼ぶ（#32の導線に相乗り）
    提案2 日報リスク    → 夜間バッチ（/api/batch/* と同じ枠）
    提案3 品質ナレッジ  → Lark Bot
    提案4 KPI/PDCA      → 月次バッチ ＋ 生産KPI画面から手動実行
    提案5 書類ドラフト  → 各画面のボタン（工事写真台帳など）
    提案6 オペレーター  → 当面見送り

■ 認証の考え方
  Lark Bot 経由の場合、「Botに話しかけた人が誰か」を必ずMembry側の社員に
  解決し、その人の権限でツールを実行する。ここを省くと
  「Botに聞けば見られない情報が見られる」という権限バイパスになる。
  lib/menu-permission.ts の getEmployeeByLarkId() がそのまま使える。

4. モデル選定
================================================================================
■ 現行と推奨

  用途                          現行              推奨
  ─────────────────────────────────────────────────────────────────────────
  エージェント中核（提案1〜4）  （なし）          claude-opus-4-8
  画像・OCR（工程表/現場写真）  claude-sonnet-4-6 claude-opus-4-8
  大量バッチの分類・要約        claude-haiku-4-5  claude-haiku-4-5（維持）
  短い総評生成                  claude-haiku-4-5  claude-haiku-4-5（維持）

■ Opus 4.8 を中核に推す理由
  ・長い手順を自分で回しきる（複数ツールを跨ぐ提案1・4に直結）
  ・画像の長辺2576pxまで対応（Sonnet 4.6は1568px）。図面・現場写真で効く
  ・1Mコンテキストを追加料金なしで使える

■ effort（思考の深さ）は用途ごとに振り分ける
    output_config: { effort: "low" | "medium" | "high" | "xhigh" | "max" }
  ・提案1・4（多段の調査と判断）      → high
  ・提案2（日報の一次スクリーニング） → medium（件数が多くコスト影響大）
  ・提案3・5（検索と下書き）          → medium
  高くすれば良いわけではない。自社の評価データで low/medium/high を
  実際に振って比べ、精度・待ち時間・コストの折り合いで決める。

■ 【要修正】既存コードの互換性
  app/api/ocr/schedule/route.ts:61
      thinking: { type: "enabled", budget_tokens: 8000 }
  この書き方は Sonnet 4.6 では非推奨ながら動くが、
  Opus 4.8 / Opus 4.7 / Sonnet 5 では 400 エラーになる。
  モデルを上げる際は同時に次へ置き換える必要がある。
      thinking: { type: "adaptive" },
      output_config: { effort: "high" }
  同様に temperature / top_p / top_k も新しいモデルでは 400 になる。
  （現状これらは未使用なので、こちらは影響なし）

■ 廃止耐性は現行設計を維持する
  AI_MODEL_CHAINS のフォールバックと /api/health/ai-models の監視は
  そのまま活かす。エージェント用のチェーンを1本足すだけでよい。
      AGENT_CORE : ["claude-opus-4-8", "claude-sonnet-5"]

5. ツール設計の指針
================================================================================
■ 既存APIをそのままツールにする
  新しくAI専用のデータ層を作らない。app/api/* が既に
  「認証・権限・Lark接続・エラー整形」を済ませているため、
  ツール関数はそれを呼ぶだけの薄い層でよい。

■ description には「いつ呼ぶか」を書く
  ✕ 「参考図台帳を検索します」
  ○ 「建屋の用途・寸法・部材条件から過去の参考図面を探すときに呼ぶ。
      ユーザーが建物の仕様を口頭で述べた時点で、まずこれを呼ぶこと」
  最近のモデルはツールを控えめにしか使わないため、
  トリガー条件を書くかどうかで呼び出し率が実際に変わる。

■ Lark へ書き戻すなら必ず構造化出力
  json_schema による structured outputs、またはツール定義の strict: true を
  使い、返り値の形を保証する。自由文をパースする実装は必ず壊れる。
  ※ 単一選択フィールドに書く値は、Lark側の選択肢と完全一致していないと
    1254302 Permission denied になる（既知の落とし穴）。
    列挙値はスキーマの enum に固定し、モデルが新しい値を作れないようにする。

■ プロンプトキャッシュを効かせる
  参考図台帳のマスタや長い業務ルールを毎回投げるなら、
  安定した内容を先頭に置き cache_control を付ける。
  キャッシュは前方一致なので、日時やユーザー名をシステムプロンプトの
  先頭に埋め込むと毎回全滅する。動的な値は必ず後ろへ置く。

■ 出典を持たせる
  「なぜそう言えるか」を必ず出させる。参考図なら図面番号、
  日報なら写真ID、KPIならKPIコードと期。
  根拠のない出力は現場で使われない。

6. ガードレール
================================================================================
  ・権限     : ツール実行前に必ず requireMenuAccess() を通す。
               「画面で見られないものは、AIにも見せない」
  ・書き込み : 当面は読み取り専用。書き込みは人の承認を挟む
  ・注入対策 : 日報・書類の本文は「データ」であって「指示」ではない。
               本文中の命令文に従わないようシステムプロンプトで明示する
  ・監査     : 誰が・いつ・どのツールを・どの引数で呼んだかを記録する
               （lib/kpi-audit.ts と同じ考え方を横展開）
  ・免責表示 : AIの出力には必ず「一次案・要確認」を明示。
               特に提案2の安全指摘は、人の確認を経ずに現場へ流さない
  ・コスト   : max_tokens とツール呼び出し回数に上限を設ける。
               件数の多い提案2は task_budget での自己制御も検討に値する

8. やらないほうがよいこと
================================================================================
 ・全社横断の巨大ベクトルDBを最初に作る
   → 構造化データが大半。検索APIをツール化するほうが正確で安価。
     RAGが要るのは非構造の長文（仕様書・議事録）に限られる。

 ・KPIの判定や原価の計算をAIにさせる
   → 決定論的な純関数がすでにある。AIは解釈だけを担当する。

 ・既存の単発呼び出し3か所を無理にエージェント化する
   → 総評生成とOCRは単発が正しい。エージェント化しても精度は上がらず、
     コストと待ち時間だけ増える。

 ・自社データを扱うエージェントをAnthropic側ホスト（Managed Agents）で動かす
   → #32でセキュリティを理由に埋め込みWeb版を撤去した判断と整合しない。
     同じ基準で考えるなら自社ホスト（Tool Runner）を選ぶべき。

 ・人の確認なしにAIの出力を現場へ流す
   → 特に安全・品質の指摘。誤検知が信頼を壊し、運用が止まる。


================================================================================
9. 確認したい事項
================================================================================
 (1) Anthropic API の利用枠・予算の想定
     エージェントは単発呼び出しより1件あたりのトークンが増える。
     どの規模まで許容できるかで、effort と対象件数の設計が変わる。

 (2) 社内AIチャット(#32)の Lark Bot は、どこまで機能追加できるか
     別プロジェクト(shainai-help-bot)側の実装次第で、
     提案1・3を「Bot に足す」か「Membry側に新UIを作る」かが変わる。
