# 06. 段階的実装計画

要件 §7.2 の進め方(モックアップ先行・会議見学・対話で詰める)を踏まえ、リスクの低い順に段階導入する。**各フェーズ末で生産本部レビュー**を挟む。

|-------|--------|---------|------|
| 0 | 要件確定・モック合意 | 画面モック / 確認事項クローズ | 〜2週 |

## Phase 0 — 要件確定・モックアップ(コーディング前)

- 各設計ドキュメント末尾の **❓要確認**(計20項目超)を生産本部・DX推進室とクローズ。
- 最優先確認:
  1. KPI実績の保持形式(縦持ち採用可否)
  2. Lv2/Lv3 実績=自動ロールアップ or 手入力
  3. 進捗率/判定の正本値(Excel全63指標)固定
  4. 6ロールと役職情報の所在
  5. 課長ロールのスコープ
- **モックアップ**: ①ダッシュボード ②実績入力 を Figma もしくは静的 page で作成し、会議投影レビュー。
- 月次レビュー会議を1回**見学**し、実運用の動線を設計に反映。

**Exit基準**: 確認事項クローズ + モック合意。

## Phase 1 — 基盤・マスタ・ダッシュボード(閲覧)

1. **Lark Base テーブル作成**(`02_data-model.md` の全テーブル ≒14本)。テーブルID を env / フォールバックに登録。
   - 経営レイヤー(3): `KEIEI_MIDTERM_PLAN_HEADER` / `KEIEI_MIDTERM_PLAN` / `KAIKEI_ACTUAL`(+ 既存 `COMPANY_KPI` 流用)
   - 生産本部レイヤー(9): `SEISAN_KPI_MASTER` / `PERIOD` / **`GROUP` / `GROUP_MEMBER`** / `ACTUAL` / `MEASURE` / `PDCA` / `STAR_ADJ` / `HISTORY` / `AUDIT`
   - ※ `SEISAN_KPI_BASIS` は作らない(`KAIKEI_ACTUAL` に統合)
2. `lib/lark-tables.ts` 拡張(`getLarkTables` / `TABLE_BASE_CONFIG` / `*_FIELDS` / `MENU_GROUPS` に `経営`・`生産本部KPI`)。
3. **部署マスタ = Lark部署**: 既存 `fetchLarkDepartmentTree`/`expandDepartmentChain` を流用。9課の `open_department_id` を確定・対応付け。
4. **データ移行スクリプト**(`scripts/`):
   - `07_KPIマスタ`(63指標)→ `SEISAN_KPI_MASTER`(部署=Lark部署ID紐付け)
   - 期マスタ(50期, 経過月数9)→ `SEISAN_KPI_PERIOD`
   - **グループ初期投入** → `SEISAN_KPI_GROUP` / `SEISAN_KPI_GROUP_MEMBER`(鉄工課G/北関東工場G 等。北関東鉄工課の重複所属含む)
   - `06_過去実績`(43-49期)→ `SEISAN_KPI_HISTORY`(部署別分は部署ID付与)
   - 50期実績(02シート)→ `SEISAN_KPI_ACTUAL`(縦持ち変換)/ 基礎データ → `KAIKEI_ACTUAL`
   - 中計(50→52期 線形補間)→ `KEIEI_MIDTERM_PLAN_HEADER`/`_PLAN`
   - 既存 `scripts/*.ts`(dump/registerパターン)を流用。
5. `lib/kpi/` 算出ユーティリティの**骨子**(集計・進捗率・判定・★・**グループ集計**・**中計線形補間**)+ ダッシュボード集約API(生産本部・経営)。
6. **ダッシュボード画面**(生産本部・経営、閲覧のみ)+ メニュー投入スクリプト(経営/生産本部KPI)。
7. メニュー権限(グループ/個別)に閲覧 program を仮設定。

**Exit基準**: 生産本部ダッシュボードが Excel `01` と一致。経営ダッシュボードが中計トラジェクトリ+全社KPIを表示。

## Phase 2 — KPI実績入力 + 判定(中核)

1. `lib/kpi/` 算出ユーティリティ**完成** + **回帰テスト**(Excel 全63指標の進捗率/判定を期待値に固定し一致検証)。
2. `/api/seisan-kpi/actuals`(部署単位 upsert + AUDIT) + **`/api/keiei/kaikei-actual`**(会計データ・粒度正規化)。
3. **実績入力画面**(入力即判定・経過月数管理) + **経営「会計データ入力」画面**(月/四半期/半期/年)。
4. AUDIT テーブルへの記録基盤。

**Exit基準**: 入力→即判定が Excel と一致。会計データ入力で粗利率等(Lv2)・全社KPI進捗が算出。任意の月編集で累計/平均/直近月値が再計算。

## Phase 3 — 施策PDCA + ★達成評価

1. `/api/seisan-kpi/measures`(グループ単位・状態管理) `…/pdca`(効果自動判定) `…/stars`(部署ごと) + `/groups`・`/groups/members`。

5. ダッシュボードの施策進捗・★ブロックを実データに接続。

**Exit基準**: ★合計が Excel `04/05` と一致(部署ごと)。施策のKPI自動表示・グループ集約・効果自動判定が動作。

## Phase 4 — 権限・ロック・エクスポート

1. **6ロール制御**(`05_` §3.2): KPIロール解決(Lark部署スコープ)+ API行レベルガード(読取/書込)。会計入力=本部長限定。
2. **月次ロック**: 確定フラグ + 管理者解除フロー + サーバ側再検証。
3. **KPIマスタ管理 / グループマスタ管理 / 中計マスタ管理**(管理者)+ **期切替**(前期複製)+ **中計の追加**(53→55期…)。
4. **過去実績参照**(全社/部署別/グループ別 3スコープ)。
5. **ヘルプ(運用ガイド)**: Excel 00 を初期コンテンツに移行、全画面ヘッダ「？」導線。
6. **エクスポート**(CSV / PDF)。
7. セキュリティ確認(ロールごとの可視/編集範囲、経営KPI・会計データの閲覧/編集制限)。

**Exit基準**: 各ロールで想定どおりの可視/編集範囲。確定データがロックされ不正変更不可。中計の追加・グループ重複所属が動作。

## Phase 5 — 並行運用・移行確定

- 1〜2ヶ月、**Excel と本システムを並行運用**し毎月の値・判定・★の一致を検証(要件 §7.1)。
- 差異は算出ユーティリティ or 移行データを補正。
- 一致確認後、Excel 運用を停止し本システムへ完全移行。
- 運用ガイド(画面版)整備、利用者トレーニング。

**Exit基準**: 並行運用で連続一致 → 本番単独運用へ。

## リスクと対策

| リスク | 影響 | 対策 |
|--------|------|------|
| 判定/算出が Excel と微妙に不一致 | 信頼失墜 | 全63指標の回帰テスト + 並行運用で吸収 |
| Lv2/Lv3 ロールアップの仕様揺れ | 数値ズレ | Phase0 で自動/手入力を確定。当面は Excel と同じ手入力で安全側 |
| データ行レベル権限が既存に無い | セキュリティ要件未達 | API側ガードを新設(§05) |
| Lark Bitable のフィルタ制約(OR/配列) | 不具合 | 既存知見(`OR()` 関数形式・全件取得検索)を踏襲 |
| Amplify SSR で env 取得不可 | 本番障害 | 既存同様フォールバック値をコードに保持 |
| 大量レコードで 504 | 会議で表示不能 | 期フィルタ + 一括取得 + キャッシュ(既存BI 504対策に倣う) |
| 組織変更(課の統廃合) | 拡張性要件 | 期マスタ + 有効フラグ + 部署選択のマスタ化で対応 |
