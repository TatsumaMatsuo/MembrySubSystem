## 1. 背景と目的

### 1.1 現状の課題

| No | 課題 | 影響 |
|----|------|------|
| 1 | 複数自治体サイトを毎日手作業で巡回し、入札情報を収集している | 工数増大、休日・繁忙期の巡回漏れ |
| 2 | 条件に合う案件を見落とし、提案機会を逃している | 受注機会損失 |
| 3 | 入札情報の収集・管理がベテラン営業担当に属人化 | 業務停止リスク、ノウハウ非継承 |
| 4 | 情報収集に時間を取られ、積算準備や提案書作成に集中できない | 提案品質・応札率の低下 |
| 5 | 案件情報が社内で共有されず、チーム間で重複対応が発生 | 二重工数、営業所間の連携不全 |
| **6** | **入札公告時点では仕様が確定しており、提案の余地がない** | **価格競争のみの勝負となり、利益率が低下。自社製品の優位性を仕様に反映できない** |
| **7** | **自治体の予算・計画情報を組織的に把握していない** | **1〜3年先の案件を予見できず、先行提案・情報提供活動ができない** |

### 1.2 目的

1. **【川下】** 入札公告情報を定期自動収集し、Lark Base に一元登録・共有する
2. **【川上】** 自治体の**発注見通し・当初予算・総合計画・実施計画・個別施設計画**等を**年2回**巡回し、入札公告より **6ヶ月〜3年前** の段階で案件の芽を捕捉する
3. 捕捉した上流情報を**提案活動リード**として営業所へ展開し、仕様形成段階でのアプローチを可能にする
4. 検索キーワードを**現場が自由に追加・調整できるマスタ**として管理し、属人化を解消する
5. 上流情報 → 入札公告 → 受注 を**同一事業として追跡**し、先行捕捉の効果を定量評価する

### 1.3 情報の時間軸（本システムの狙い）

```
     ▼総合計画・個別施設計画      ▼当初予算案   ▼発注見通し    ▼入札公告   ▼開札
─────┼──────────────────────────┼────────────┼─────────────┼──────────┼─────→
     │                           │            │             │          │
   3〜5年前                   1〜3ヶ月前     年度当初      1〜2ヶ月前
     │←─────── 提案・仕様形成が可能な期間 ──────→│←─価格競争のみ─→│
     │                                          │
     │        ★本システムで新たに捕捉する領域    │  ← v1.0 の対象範囲
     └── 川上収集：春（4〜5月）／秋（10〜11月）の年2回でこの領域をスキャン ──┘
```

### 1.4 定量目標（KPI 案）

- 情報収集作業時間：現状比 80% 削減
- 入札公告の条件合致案件 捕捉率：95% 以上
- **上流段階での案件捕捉件数：年間 100件以上**（年2回の収集で捕捉）
- **先行捕捉率（入札公告案件のうち上流で事前捕捉できていた割合）：初年度 30% → 3年後 60%**
- 公告掲載から社内共有までのリードタイム：24時間以内
- **川上収集の実施率：年2回とも予定月内に完了（100%）／各回で対象自治体の未更新チェック完了**

## 2. スコープ

### 2.1 対象範囲

**【川下：入札公告系】**
- 自治体・官公庁の入札情報公開サイトからの公告情報の自動収集（**日次**）
- 正規化・重複排除・条件マッチング・Lark Base 登録
- 営業事務による要否判定、営業所チャットルームへの通知
- 案件進捗管理・分析

**【川上：予算・計画系】（v1.1 追加 / v1.2 で年2回運用に設計変更）**
- 公共工事の**発注見通し**（年度当初・年内更新）
- **当初予算案／予算概要／予算説明資料**（主要事業一覧）
- **総合計画**（基本構想・基本計画・**実施計画**）
- **公共施設等総合管理計画／個別施設計画**（長寿命化計画・再編計画）
- **各種分野別計画**（防災計画、スポーツ施設整備計画、都市計画マスタープラン、脱炭素・EV関連計画 等）
- **パブリックコメント**、**議会 議案・議事録**（補正予算での新規事業）※随時公表分は原則手動臨時実行で捕捉
- 上記からの事業抽出、提案活動リード管理、事業追跡
- 収集頻度：**年2回（春・秋）の一括バッチ＋必要時の手動臨時実行**

**【共通】**
- キーワードマスタの現場運用（追加・編集・効果測定）
- MembrySubSystem（Webアプリ）からの起動・参照

### 2.2 対象外

- 電子入札システムへの実際の応札操作
- 積算・見積計算そのもの
- 民間工事の引合情報
- 提案書・技術資料の自動生成（将来的に Menkuma 連携で検討）
- 川上情報の**常時（日次・月次）監視**（年2回のバッチと手動臨時実行に限定）

## 4. 機能要件（川下：入札公告系）

### F-01 収集対象サイト管理

| 項目 | 内容 |
|------|------|
| 概要 | 巡回対象サイトを Lark Base のマスタで管理する |
| 登録項目 | サイトID／自治体名／都道府県／URL／**情報種別（入札公告／発注見通し／予算／計画／議会／パブコメ）**／サイト種別（PPI型・独自型・PDF型）／パーサID／**収集タイミング（川下=日次／川上=春・秋・両方・手動）**／巡回時刻／有効フラグ／最終成功日時／担当営業所 |
| 要件 | ・サイト追加はマスタ登録＋パーサ設定のみで対応可能な構成<br>・有効フラグ OFF で一時停止<br>・巡回失敗が N 回連続でアラート通知 |

### F-02 定期巡回・情報取得（入札公告）

| 項目 | 内容 |
|------|------|
| 実行タイミング | 毎営業日 早朝（例：06:00）。サイト単位で頻度変更可 |
| 取得方式 | HTML一覧パース／詳細ページ取得／添付ファイル（PDF・Excel・Word）のDLとテキスト抽出（OCR含む） |
| 負荷配慮 | サイトごとにアクセス間隔（3秒以上）を設定。robots.txt・利用規約を遵守。User-Agent に自社名・連絡先を明示 |
| 差分取得 | 取得済み公告URL・公告番号を保持し新規分のみ処理 |
| リトライ | 通信エラー時 最大3回。失敗時はログ記録＋アラート |

### F-03 データ正規化・重複排除

| 項目 | 内容 |
|------|------|
| 正規化 | 和暦→西暦、金額の数値化（千円・百万円単位の統一）、全角半角統一、入札方式のコード化 |
| 重複判定 | 発注機関コード＋公告番号を主キー。番号なしは「発注機関＋案件名＋開札日」のハッシュ |
| 更新検知 | 同一キーで内容差分があれば履歴を残し変更通知（入札日変更、公告取消 等） |

### F-04 キーワードマッチング／スコアリング【全面改訂（v1.1）】

#### 4-1 設計方針

キーワードは**システム改修なしに現場が無制限に追加できる**ことを最重要要件とする。ハードコードは一切行わない。

#### 4-2 キーワードの構造

| 階層 | 内容 | 例 |
|------|------|-----|
| **適用ソース** | どの情報種別に適用するか | 入札公告／発注見通し／予算／計画／全て |
| **カテゴリ** | キーワードの意味分類 | 製品系／用途系／工種系／状態系／除外系／地域／発注機関 |
| **キーワード** | 実際の検索語 | 膜構造、テント倉庫、上屋、屋根改修 |
| **同義語・表記ゆれ** | 同一扱いする語群 | テント／天幕／幕体、上屋／うわや |
| **重み** | スコア加算値（1〜10） | 「膜構造」=10、「倉庫」=2 |
| **判定条件** | 単独ヒットで要対応か、組合せ必須か | 「倉庫」は「新設」「整備」との AND 条件 |

#### 4-3 カテゴリ別 初期登録キーワード（例・随時追加）

| カテゴリ | キーワード例 |
|---------|-------------|
| 製品系（高重み） | 膜構造、膜屋根、テント、天幕、幕体、エアドーム、シェルター、上屋、テント倉庫、開閉式屋根 |
| 用途系 | 倉庫、備蓄倉庫、防災倉庫、格納庫、駐輪場、駐車場上屋、屋外ステージ、観覧席、休憩所、日除け、シェード、荷捌き場、選別場、堆肥舎、農業用ハウス |
| 施設系 | 体育館、屋内運動場、多目的広場、球場、プール、公園整備、道の駅、避難所、給食センター |
| 工種・行為系 | 新設、新築、整備、改修、増築、更新、長寿命化、老朽化対策、建替、屋根改修、防水改修 |
| 環境・新分野系 | EV充電、充電設備、カーポート、太陽光、脱炭素、ZEB、再生可能エネルギー ※[[MEMBRANE EV PORT]] 関連 |
| 除外系 | 舗装のみ、電気設備単独、樹木剪定、清掃業務、警備業務、測量、設計業務のみ |

#### 4-4 マッチング処理

1. 対象文書のテキストから、有効フラグONのキーワードを全件走査
2. 同義語も同一キーワードのヒットとして扱う
3. 除外系キーワードのヒットはスコアを減算（または即時「対象外候補」）
4. 重み合計＝**マッチスコア**
5. しきい値（マスタで設定可）以上 → 「要判定」、未満 → 「参考」として登録
6. ヒットしたキーワードと該当箇所（前後100文字）を記録し、判定者が根拠を確認できるようにする

#### 4-5 AI 補助判定（推奨）

キーワードで拾えない案件を補完するため、社内LLM（[[Menkuma]]）に本文を渡し、以下を出力させる。

- 該当性判定（該当／非該当／要確認）
- 判定理由（100字程度）
- 想定される自社製品カテゴリ

キーワードスコアと AI 判定の**両方**を Base に保持し、営業事務が最終判断する（AI に判断を委ねない）。

#### 4-6 キーワードの運用と効果測定

- 追加・編集は Lark Base 上、または MembrySubSystem の「キーワード保守」画面から実施
- 権限：営業事務・営業部が編集可（DX推進室の改修不要）
- **ヒット実績を自動集計**し、以下を月次レポートで可視化
  - キーワード別ヒット件数／うち「要対応」判定率（＝有効度）
  - ヒットゼロが継続しているキーワード（棚卸候補）
  - 「対象外」理由に「工種不一致」が多いキーワード（重み見直し候補）
- 「対象外」判定された案件の本文から**未登録の頻出語を自動抽出**し、キーワード候補として提示する（半自動での辞書成長）

> 川上文書のキーワード効果は、収集が年2回のため**回別（春・秋）に集計**する。件数母数が日次の川下より小さい点に留意（棚卸は年1〜2回で十分）。

### F-05 Lark Base 登録
（v1.0 から変更なし）収集・判定済み案件を T2 へ自動登録。初期ステータス「未判定」。添付原本は S3 保存、Base にリンク保持。実行結果を T6 に記録。

### F-06 要否判定（営業事務トリアージ）

| 項目 | 内容 |
|------|------|
| 実施者 | 営業事務 |
| 判定区分 | 要対応／対象外／保留／重複 |
| 入力項目 | 判定区分、担当営業所、対象外理由（選択式）、コメント |
| 要件 | 判定者・日時を自動記録／「要対応」で F-07 発火／未判定リマインド／参加申請期限が近い案件を強調表示／**上流情報で既に追跡中の事業があれば紐付け候補を自動表示（F-19）** |

### F-07 営業所チャットルーム通知
Lark Bot によるカード送信。宛先は T5 営業所マスタの Webhook。案件名／発注機関／入札方式／申請期限／開札日／予定価格／履行場所／ヒットキーワード／公告URL／Base詳細リンク／**上流情報の有無**を含む。

### F-08 進捗管理

```
情報登録 → 要否判定 → 参加検討 → 参加申請 → 積算中 → 入札済 → 落札 → 受注(工事番号発番)
                          └→ 見送り        └→ 失注
```

案件 × 担当営業所で管理。重複着手時は警告し主担当を確定。受注確定時に工事番号・売約伝票と紐付け（[[sales-support-tool]] 連携）。

## 5. 機能要件（川上：予算・計画系）【v1.1 新設 / v1.2 で年2回運用に変更】

### 5.0 川上情報収集の基本設計（年2回）【v1.2 追加・本版の主眼】

川上情報の収集・解析は、公表実態に合わせ**原則 年2回のバッチ**で実施する。日次・月次の常時巡回は行わない。

#### 実行タイミング（EventBridge の固定 cron）

| 回 | 実行月（目安） | 主対象 | 狙い |
|----|--------------|--------|------|
| **第1回：春** | **4月下旬〜5月** | 当年度の**当初予算（確定版）**、**発注見通し（初版）**、年度初に更新される実施計画・個別施設計画 | 当年度の発注予定を確定情報で捕捉 |
| **第2回：秋** | **10〜11月** | **上半期の補正予算**、**発注見通しの年内更新**、**次年度予算の編成・概算要求動向**、秋改訂の各種計画 | 次年度案件の先行捕捉、アプローチ時期の起点づくり |

> 実行月は自治体の公表実態に合わせ調整可能（§12 Q22）。各回は対象自治体を数日かけて順次処理する（1文書10分以内・サイト単位で独立実行）。

#### 手動臨時実行（運用安全弁）

大型補正予算・注目自治体の計画公表など、年2回の枠外で必要になった場合に**人手で起動**できる臨時実行を用意する。これは自動巡回頻度を上げるものではなく、あくまで例外対応。実行単位は「自治体×情報種別」を指定可能とする。

#### トレードオフ（割り切り）

- 発注見通しの四半期更新（7月・1月）、随時公表のパブコメ・議会議案などは、**原則この年2回では即時捕捉しない**。
- 緊急性が高い種別は「手動臨時実行」または川下（入札公告）側での捕捉に委ねる。
- この割り切りの許容範囲・例外種別は §12 Q21 で確認する。

### F-13 上流情報ソース管理

自治体ごとに、情報種別・公表時期・**どの収集回で取るか**を管理する。

| 情報種別 | 主な内容 | 一般的な公表時期 | **収集回** | 先行期間 |
|---------|---------|----------------|-----------|---------|
| **発注見通し** | 年間の発注予定工事一覧（工事名・概要・種別・時期・規模） | 年度当初（4月）＋年内更新 | **春・秋（両方）** | 3〜12ヶ月 |
| **当初予算案・予算概要** | 主要事業・新規事業の一覧と予算額 | 1〜3月（議会前後） | **春**（確定版を4〜5月に収集） | 6〜18ヶ月 |
| **補正予算** | 年度途中の追加事業 | 6・9・12・3月議会 | **秋**（上半期分）＋手動（下半期分） | 3〜12ヶ月 |
| **総合計画（実施計画）** | 3〜5年の事業計画。ローリング方式で毎年更新 | 年度末〜年度初 | **春**（＋秋に差分） | 1〜3年 |
| **公共施設等総合管理計画／個別施設計画** | 施設ごとの改修・更新・統廃合の予定と時期 | 随時改訂 | **春・秋（差分収集）** | 1〜5年 |
| **分野別計画** | 防災、スポーツ施設、都市計画、脱炭素 等 | 随時 | **春・秋（差分収集）** | 1〜5年 |
| **パブリックコメント** | 計画案の意見募集（＝計画確定前） | 随時 | **手動臨時**（即時性が高いため） | 1〜3年 |
| **議会 議案・議事録** | 補正予算、債務負担行為、新規事業の議論 | 議会会期後 | **秋**＋手動 | 3〜18ヶ月 |

> **優先度の考え方**：即効性は「**発注見通し**」が最も高く、次いで「当初予算」です。総合計画・個別施設計画は先行期間が長い分、提案の自由度が高くなります。**Phase 2 では発注見通しと当初予算の2種に絞って着手**することを推奨します。

**マスタ登録項目**：ソースID／自治体名／情報種別／掲載ページURL／想定公表時期／**収集回（春／秋／両方／手動）**／ファイル形式（HTML/PDF/Excel）／パーサID／最終取得日時／最終更新検知日／有効フラグ／担当営業所

### F-14 上流情報の巡回・取得

| 項目 | 内容 |
|------|------|
| 巡回方式 | **年2回のバッチ**で掲載ページを走査し、**前回実行以降に更新（新規掲載・差替え）された文書のみ**後続処理を起動 |
| 更新検知 | 文書URL・ファイル名・更新日時・ファイルハッシュを**前回収集回の記録と比較**（常時監視ではなく実行時点での差分判定） |
| 取得対象 | PDF（大半）、Excel（発注見通しは Excel/CSV 公開が多い）、HTML |
| テキスト抽出 | PDF はテキスト層優先、なければ OCR。表構造は表として抽出（発注見通しは表形式が中心） |
| 実行スケジュール | **固定の年2回（春・秋）**。実行月は EventBridge で設定（§5.0）。公表変動が大きい自治体は手動臨時実行で補う（自動頻度は上げない） |
| 未掲載検知 | **各収集回で**、想定公表時期を過ぎても更新がない自治体をリスト化し、手動確認を促す |

### F-15 文書解析・事業抽出

長大なPDF（数十〜数百ページ）から、当社に関係し得る**事業単位**を抽出する。

**処理フロー**

1. 文書をセクション／表単位に分割
2. F-04 のキーワードマスタで一次スクリーニング（該当ページのみ抽出）
3. 該当箇所を社内LLM（[[Menkuma]]）に渡し、以下の項目に**構造化**

| 抽出項目 | 内容 |
|---------|------|
| 事業名 | 例：〇〇小学校屋内運動場改築事業 |
| 所管部署 | 教育委員会 施設課 等 |
| 事業概要 | 200字程度の要約 |
| 予算額・事業費 | 年度別内訳があれば併記 |
| 実施予定年度 | 単年度／複数年度 |
| 施設種別・所在地 | 市区町村レベル |
| 事業段階 | 構想／設計／工事／継続 |
| 当社関連度 | 高／中／低 ＋ 判定理由 |
| 想定製品 | 膜構造／テント倉庫／上屋／EVポート 等 |

4. 抽出結果を T7（上流情報）に登録。原本PDFの該当ページへのリンクを保持
5. 関連度「高」「中」を営業事務のレビュー対象とする

> **重要**：LLM の抽出結果は必ず**原本の該当箇所へのリンクとセットで保存**してください。予算額や年度の誤読は判断を誤らせます。営業事務のレビュー時に原本を1クリックで確認できる導線が必須です。
>
> **年2回化の効果**：解析対象が年2回に集中するため、LLM 利用コストは**回別に見積り・上限管理**でき予見性が高い。反面、1回あたりの処理量が大きくなるため、各回は数日のバッチウィンドウを確保する（§8 性能）。

### F-16 上流情報のレビューと展開

| 項目 | 内容 |
|------|------|
| 実施者 | 営業事務（一次）→ 営業所（二次判断） |
| 実施時期 | **各収集回（春・秋）の直後にレビュー会を実施**（§9 半期サイクル） |
| レビュー区分 | 提案対象／情報共有のみ／対象外／継続監視 |
| 展開先 | 「提案対象」確定時に担当営業所チャットへ通知。**入札公告と区別できるカードデザイン**（例：「【先行情報】」ラベル） |
| 通知内容 | 事業名／自治体・所管課／事業費／実施予定年度／概要要約／原本リンク／推定アプローチ時期 |
| アプローチ時期の算出 | 実施予定年度から逆算し「予算要求期（前年度8〜10月）」「設計発注前」等の推奨アプローチ時期を自動提示 |

### F-17 提案活動リード管理

上流情報を起点とした営業活動を記録・追跡する。**収集は年2回だが、提案活動そのものは通年で継続**する（リマインドは日次で稼働）。

**ステータス遷移**

```
情報捕捉 → レビュー → 提案対象 → 初回接触 → 情報提供/提案 → 仕様反映 → 入札公告待ち → 応札 → 受注
                        └→ 継続監視（時期尚早）        └→ 対象外
```

| 管理項目 | 内容 |
|---------|------|
| 活動記録 | 訪問日／面談者（所属・役職）／提案内容／反応／配布資料 |
| 次アクション | 次回アプローチ予定日、担当者 |
| リマインド | 予定日の3日前に担当者へ通知。**推奨アプローチ時期の1ヶ月前に自動リマインド**（通年・日次で判定） |
| 重複防止 | 同一自治体・同一事業に複数営業所が接触しようとした場合に警告 |
| 蓄積効果 | 自治体別の接触履歴が営業所間で共有され、担当交代時に引継ぎ可能 |

### F-18 自治体カルテ（推奨機能）

自治体単位で情報を集約したビューを提供する。

- 基本情報（人口、財政規模、担当営業所）
- 過去の当社受注実績・応札履歴・落札率
- 現在追跡中の上流事業一覧
- 直近の接触履歴
- 主要な計画文書のリンク集
- 次年度以降の想定案件（金額・時期）
- **直近の川上収集回（春／秋）と、その回で検知した更新の有無**

営業所単位・全社単位でのパイプライン把握に直結します。

### F-19 事業追跡・名寄せ

**上流情報 → 入札公告 → 受注**を同一事業として串刺し管理する。

| 項目 | 内容 |
|------|------|
| 名寄せキー | 自治体コード ＋ 施設名／事業名の類似度（表記ゆれを吸収） |
| 判定方式 | 文字列類似度＋LLMによる同一事業判定の併用。**自動確定はせず候補提示 → 人が確定** |
| 効果 | 入札公告時に「この案件は昨年から追跡中」「〇月に△△課へ提案済み」が即座に分かる |
| 分析 | **先行捕捉率**（入札公告のうち上流で捕捉済みだった割合）、先行捕捉案件の落札率 vs 非捕捉案件の落札率 |

> この比較分析が、上流情報収集への投資対効果を経営層に示す最も説得力のある材料になります。Phase 2 の設計時点で必ず計測できる形にしておいてください。

## 8. 非機能要件

| 分類 | 要件 |
|------|------|
| 可用性 | 日次・バッチが失敗しても再実行で回復可能。手動再実行機能を用意。**川上は年2回のため失敗の影響が大きい（次回まで半年）。各回の完了を人が必ず確認し、失敗時は当該回の期間内に再実行する** |
| 性能 | 入札公告の全サイト巡回を1回60分以内。**川上は年2回のバッチで、1文書あたり10分以内に解析完了。1回あたりの処理量が大きいため数日のバッチウィンドウを確保**。サイト単位で独立実行しタイムアウトが全体を止めない |
| 拡張性 | サイト追加はマスタ登録＋パーサ設定のみ。**キーワード追加はマスタ登録のみでシステム改修不要**。100サイト規模まで拡張可能 |
| 保守性 | パーサはサイト種別ごとに共通化。PPI型は共通パーサで多数自治体をカバー |
| LLM利用 | 解析対象は該当ページのみに絞り、トークン消費とコストを抑制。**川上解析は年2回に集中するため回別に上限設定・実績監視**。月次では川下のAI補助判定分を監視 |
| セキュリティ | AWS 上で実行、認証情報は Secrets Manager。Base のアクセス権限は職務単位。参照用Web画面はメニュー権限のサーバ側強制ガードに従う（[[menu-access-server-enforcement]]） |
| ログ・監査 | 全実行ログを90日以上保持。判定・レビュー操作の履歴を保持。**川上は各収集回の実行結果（T6）を保持し、回次比較に利用** |
| 監視 | 巡回失敗、取得件数ゼロ、必須項目欠落を検知しDX推進室へ通知。**川上は各収集回で「想定公表時期の未更新」を検知しリスト化** |
| データ保持 | 公告・計画文書の原本は S3 に**5年保持**（複数年計画の追跡に必要） |

| 頻度 | 内容 |

## 10. 法務・コンプライアンス上の留意点

- 収集対象は**公開されている情報**に限定する（入札公告、予算書、計画書はいずれも公開資料）
- 各サイトの robots.txt および利用規約を事前確認し、自動取得を禁止しているサイトは対象外とする（除外理由をマスタに記録）
- アクセス頻度を抑制し、対象サイトの運用に支障を与えない（**川上は年2回のため恒常的な負荷は小さい**）
- ログイン・会員登録が必要な領域は原則対象外
- 取得データは社内利用に限定し、外部提供・再配布は行わない
- **計画文書・予算書には自治体ごとに転載条件（政府標準利用規約準拠等）が定められている場合があります。社内利用の範囲を超える二次利用は行わないこと**
- **提案活動においては、入札公正性を損なう働きかけと受け取られないよう、通常の営業活動・情報提供の範囲を明確にした社内ルールを併せて整備してください**（この点は営業部門と法務での方針決定が必要です）

## 11. 段階的導入計画

| フェーズ | 内容 | 想定期間 |
|---------|------|---------|
| **Phase 0** | 対象サイト洗い出し・優先順位付け、現行手作業のヒアリング、**キーワード初期セットの整備**（ベテラン営業へのヒアリング）、**川上の収集回（実行月）の確定** | 2週間 |
| **Phase 1** | 【川下】収集基盤構築（優先5〜10サイト）＋Base登録＋要否判定＋チャット通知＋キーワードマスタ運用開始 | 6〜8週間 |
| **Phase 2-a** | 【川上】**発注見通し・当初予算**の2種に絞って収集開始（優先自治体10〜20）。**年2回運用を前提**に文書解析（F-15）、T7・T9 の運用開始。まず**春回相当を1〜2自治体で PoC** | 6〜8週間 |
| **Phase 2-b** | 入札側の進捗管理（T4）、MembrySubSystem（Web）連携、期限リマインド | 4週間 |
| **Phase 3** | 総合計画・個別施設計画・議会情報へ拡張、事業追跡（F-19）、自治体カルテ（F-18）、分析ダッシュボード | 6〜8週間 |

**推奨アプローチ**：Phase 2-a は**1〜2自治体の発注見通しで PoC** を行ってください。発注見通しは Excel 形式が多く構造化しやすいため、上流情報収集の中で最も費用対効果が高く、成果も早く出ます。**年2回運用のため、PoC は次の収集回（春または秋）に合わせて計画**し、実データで1回分を通しで検証してから予算書・計画書へ広げるのが安全です。
