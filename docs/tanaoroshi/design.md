## 1. 設計方針（要件からの主要な導出）

要件を素直に実装すると破綻する箇所が3点ある。本設計はそこを起点に組み立てる。

| # | 要件上の論点 | 設計判断 |
|---|---|---|
| 1 | F-02「重複読取は加算」＋ F-03「取消・修正」＋ F-06「1操作ごとの個別レコード保持」＋ 監査性 | **棚卸実績テーブルは追記専用（append-only）**。加算＝レコードを足す。取消＝元レコードの状態を「取消」に更新（物理削除しない）。修正＝取消＋新規登録。実棚数量は常に「状態=有効の SUM」で導出する。累計値をどこにも保持しない |
| 2 | F-12「一時保存・再送信」で二重送信が起きると棚卸そのものが無意味になる | **クライアント採番の `実績ID`（UUID）による冪等送信**。サーバは送信バッチ受信時に既存 `実績ID` を除外してから登録し、確定した ID を返す。ネットワーク断・二重タップ・ブラウザ再起動のいずれでも二重計上しない |
| 3 | 非機能「読取レスポンス1秒以内」＋ 倉庫内の通信不安定 | **読取のたびにサーバへ問い合わせない**。作業開始時に対象倉庫の品目情報を一括ダウンロードして IndexedDB にキャッシュし、読取判定・品目表示・未報告リストはすべてローカルで完結させる。サーバ通信は「起動時の一括取得」と「バックグラウンドの一括送信」の2回のみ |

副次的な方針:

- **オフラインファースト**: 通信断でも作業が止まらないことを最優先。送信は常に非同期・バックグラウンド。
- **既存流儀の踏襲**: Lark Base アクセスは `lib/lark-client.ts`、テーブルID/列名は `lib/lark-tables.ts` に集約、権限は `lib/menu-access.ts`、クライアント fetch は `lib/fetch-json.ts`。新しい流儀を持ち込まない。
- **Amplify 28秒タイムアウト対策**: 書込は必ず `batchCreateBaseRecords` / `batchUpdateBaseRecords` で集約。逐次ループ禁止（既知の障害パターン）。

#### 倉庫マスタ `tblYtGsHvlKEWRF3` / 品目マスタ `tblw1oc3PD0cM0fu`

**いずれも project base のテーブル一覧（62件）に存在せず、直接アクセスも `1254302 The role has no permissions` で失敗**。別 Base にあるか、アプリのロールから隠されている（残課題 R-1'）。

ただし **システム在庫情報が倉庫コード・倉庫名・品番・品名・品名2・単位をすべて保持している**ため、以下の設計とする:

- **倉庫一覧 = システム在庫情報の 最新締日 × 倉庫コードの DISTINCT**（マスタ参照不要）
- **品目情報 = システム在庫情報の該当行**（マスタ参照不要）
- 品目マスタが要るのは「システム在庫に載っていない品目を読み取った場合」の品名解決のみ。要件 F-02 では**マスタ未登録＝エラーで登録しない**方針のため、v1 ではシステム在庫にない品番を「対象外」として扱えば成立する

→ **倉庫マスタ・品目マスタへの権限取得は v1 のブロッカーではない**。ただし「システム在庫に無いが現物がある」ケースの扱いは業務確認が必要（残課題 R-9）。

#### 棚卸在庫情報 `tbl8pCg48KRx8710`（project base）— 基幹連携の出力先

**実査で判明した最重要事項。** 「入力した棚卸情報」用として既に用意されており、現在 0 件・11列。列構成が基幹システムの棚卸取込レイアウトそのものと見られる。

| 列名 | 型 | 棚卸アプリからの投入内容 |
|---|---|---|
| 倉庫コード | Number | 確定値の倉庫 |
| 倉庫 | Text | 倉庫名 |
| 品番 | Text | 品目コード |
| 品名 / 品名2 | Text | システム在庫情報から転記 |
| 数量 | Number | **確定実棚数量**（最大回数の実績） |
| 理論数 | Number | システム在庫数 |
| 差異数 | Number | 数量 − 理論数 |
| 担当者コード / 担当者 | Text | 最終入力者 |
| 備考 | Text | 差分理由・在庫状態内訳 |

**設計への反映（F-11 の方式変更）**

要件 F-11 は「CSVファイル出力」としていたが、**受け皿テーブルが既に存在するため、第一の出力方式を「このテーブルへの書き戻し」に変更**する。CSV出力は副次的な手段として残す（基幹側が Base 参照でなくファイル取込の場合に備える）。

- 出力実行時に、対象期・倉庫の既存行を削除 → 確定値を `batchCreateBaseRecords` で投入（再実行の冪等性を確保）
- この 11 列に写真・回数・在庫状態フラグは入らないため、**監査用の生ログは別テーブル `TN_ENTRY` が必須**（下記 3.2）

→ 残課題 R-7（基幹取込レイアウト）は大部分が解消。残るのは「基幹側がこのテーブルを直接読むのか、CSVを介するのか」の確認のみ。

### 3.2 新規テーブル

#### (1) 棚卸期 `TN_PERIOD`

棚卸イベントの単位。要件に明示はないが、**これが無いと「回数」と「実績」がどの棚卸のものか特定できず、次回棚卸でデータが混ざる**ため必須と判断した。

| 列名 | 型 | 内容 |
|---|---|---|
| 期ID | テキスト | `TN-YYYYMM-nn` |
| 棚卸名称 | テキスト | 例: 2026年下期 棚卸 |
| 基準日 | 日付 | システム在庫のスナップショット基準日 |
| 状態 | 単一選択 | 準備中 / 実施中 / 締め |
| 作成者 / 作成日時 | テキスト / 日付 | |

同時に「実施中」にできる期は1つのみ（アプリ側で保証）。

#### (2) 棚卸倉庫進捗 `TN_WH_STATUS`

倉庫ごとに回数が進むため、回数は期ではなく**倉庫単位**で持つ。F-08の対象制御と F-09 のダッシュボードの基盤。

| 列名 | 型 | 内容 |
|---|---|---|
| 期ID | テキスト | |
| 倉庫コード / 倉庫名 | テキスト | |
| 現在回数 | 数値 | 1〜3。入力受付中の回数 |
| ステータス | 単一選択 | 未着手 / 実施中 / 発行処理中 / 1回目確定 / 2回目実施中 / 2回目確定 / 3回目実施中 / 締め |
| 対象品目数 / 報告済品目数 / 差分件数 | 数値 | ダッシュボード用のキャッシュ値。発行時と定期集計時に更新 |
| 最終報告日時 / 更新日時 | 日付 | |

「発行処理中」は差分リスト発行の**二重実行防止ロック**として使う（Lark Base にトランザクションが無いため、状態列による楽観ロックで代替）。

#### (3) 棚卸実績 `TN_ENTRY`（追記専用・本機能の中核）

読取・入力1操作 = 1レコード。物理削除しない。

| 列名 | 型 | 内容 |
|---|---|---|
| 実績ID | テキスト | **クライアント採番のUUID**。冪等キー |
| 期ID | テキスト | |
| 倉庫コード / 倉庫名 | テキスト | |
| 品目コード / 品目名 | テキスト | 品目名は登録時点のスナップショット |
| 入力数量 | 数値 | この1操作で入力した数量（累計ではない） |
| 在庫状態 | 単一選択 | 良品 / 不良品 / 滞留（既定=良品） |
| 写真 | 添付 | 複数可 |
| 入力方式 | 単一選択 | 読取 / 手入力 / 検索 |
| 棚卸回数 | 数値 | 1〜3 |
| 差分理由コード | テキスト | 2回目以降に付与（F-08） |
| 状態 | 単一選択 | 有効 / 取消 |
| 取消元実績ID | テキスト | 修正時に、新レコードから旧レコードを指す |
| 入力者 / 入力者メール | テキスト | セッションから |
| 入力日時 | 日付 | クライアントの入力時刻（epoch ms 書込）。オフライン時は送信時刻とズレるため**入力時刻を採用** |

| 端末ID | テキスト | IndexedDB に保存する端末固有ID。障害調査用 |

**実棚数量の算出**: `期ID + 倉庫コード + 品目コード + 棚卸回数` でグループ化し、`状態=有効` の `入力数量` を SUM。累計値はどこにも保持しない（唯一の真実はこのテーブル）。

**単一選択の選択肢は Lark UI で事前に手動作成**すること。アプリから未登録の選択肢値を書き込むと `1254302 Permission denied` になる（既知の落とし穴）。

初期値: 出荷未計上 / 入庫未計上 / 破損 / 紛失 / 数え間違い / その他（R-3で最終確定）。

### 3.3 日付列の扱い

Lark の日付フィールドは JST 真夜中を UTC で保持する仕様があり、filter 文字列での日付数値比較は不安定（既知）。本機能では **epoch ms を書き込み、期間絞り込みは JS 側で行う**（既存 GANTT_CHART_FIELDS と同方式）。

## 4. オフライン設計（F-12）

### 4.1 端末側ストレージ

localStorage は 5〜10MB 上限で写真 Blob を保持できないため **IndexedDB** を使う。DB名 `tanaoroshi`、`lib/tanaoroshi/local-store.ts` に素の IndexedDB ラッパを実装（新規依存は追加しない）。

| ストア | キー | 内容 |
|---|---|---|
| `session` | `current` | 期ID、倉庫コード、回数、端末ID、開始日時 |
| `catalog` | 品目コード | 対象品目（品目コード/品目名/単位/システム在庫有無/対象フラグ）。起動時に一括DL |
| `queue` | 実績ID | 未送信の入力（写真 Blob 含む） |
| `sent` | 実績ID | 送信確定済みID（30日でパージ）。再送時の除外に使う |

### 4.2 送信フロー

```
[決定次へ] → queue へ put（この時点で入力は保全される）
              ↓
        送信ワーカー（onLine 監視 / 5秒デバウンス / 最大20件バッチ）
              ↓
      ① 写真あり → POST /api/tanaoroshi/photo（1枚ずつ octet-stream）→ file_token
      ② POST /api/tanaoroshi/entries { entries: [...] }
              ↓
      サーバ: 既存 実績ID を検索して除外 → batchCreateBaseRecords
              ↓
      レスポンス { accepted: [実績ID...], duplicated: [実績ID...] }
              ↓
      accepted ∪ duplicated を queue から削除、sent へ移動
```

- **冪等の要**: `duplicated` も削除対象に含める。これによりレスポンス欠落時の再送でも二重登録にならない。
- 送信失敗時は queue に残し、指数バックオフで再試行。失敗が5回続いたら画面に警告表示。
- 未送信件数はステータスバーに常時表示（F-12）。0件になるまで「終了」時に警告する。

### 4.3 写真の扱い

既存の `/api/documents/upload` は案件書庫用（製番必須）のため流用せず、`/api/tanaoroshi/photo` を新設する。方式は既存と同じ **`application/octet-stream` 生バイナリ + メタデータはクエリ文字列**（Base64 は膨張して Amplify Lambda の約6MB本文上限に当たるため）。

- クライアントで `compressImageToLimit(file, 1.5MB)` により圧縮（`lib/document-upload.ts` を再利用）。棚卸写真は状況記録用途のため 1.5MB で十分。
- サーバは Lark Drive へアップロードし `file_token` を返す。実績レコード登録時に添付列へ設定。
- 通信断時は Blob を IndexedDB に保持したまま、実績本体と一緒に送信待ちにする。

## 5. バーコード読取設計（F-02）

### 5.0 【要件との相違】実物は 2次元ではなく 1次元 Code 39

現品ラベルの実物サンプル（2026-07-21 提供）を確認した結果、**貼付されているのは QR / DataMatrix などの2次元コードではなく、1次元バーコードの Code 39** だった。

```
┌────────────────────────────────────┐
│ 品目        │ F00001 │ ×メッキ TCボルト │
│ 規格        │ M20×50                  │
│ 入荷日/単位  │ 2026/07/21    │  本      │
│ 備考        │                         │
├────────────────────────────────────┤
│   ‖│‖││‖│‖‖││‖│‖││‖│‖‖│‖││‖      │
│         * F 0 0 0 0 1 *              │  ← アスタリスクで囲まれている
└────────────────────────────────────┘
```

**判定根拠**: 可読文字が `*F00001*` とアスタリスクで囲まれている。これは Code 39 の start/stop 文字であり、Code 39 の決定的な特徴。バーも太細2種で構成された線形コード。

要件定義書「前提条件5：2次元バーコード」は実態と異なるため、**本設計では Code 39（1次元）を前提とする**。要件定義書の改訂を推奨（残課題 R-12）。

**設計への影響**

| 影響 | 内容 |
|---|---|
| ライブラリ | ZXing は Code 39 に対応しているため選定は変わらない。ただし読取フォーマットを Code 39 に限定する |
| **読取難度が上がる** | 1次元は2次元よりカメラ読取に弱い（斜め・距離・反射・手ブレに敏感、誤り訂正が無い）。**UI側の作り込みが2次元前提より重要になる** |
| **誤読リスク** | Code 39 はチェックディジットが任意で、このラベルには付いていない。部分読み取り（truncation）による誤読がありうる → **書式検証で防ぐ（§5.3）** |
| ラベル運用 | ラベルは「入荷日」単位＝**入荷ロットごとに貼付**されている。同一品目が複数ラベルで存在するため、要件 F-02 の「同一品目の複数読取は加算」がまさにこの運用に対応する |
| バーコードの内容 | `F00001` = 品目コードのみ。要件の前提条件4と一致（規格・入荷日はコードに含まれない） |

### 5.1 ライブラリ選定

| 候補 | 判定 |
|---|---|
| `BarcodeDetector`（ブラウザ標準） | Android Chrome では `code_39` を高速に読める。**iOS Safari 未対応** |
| **`@zxing/browser` + `@zxing/library`** | iOS Safari を含めて動作し Code 39 対応。**採用** |
| `html5-qrcode` | 内部は ZXing。UI込みで手軽だが Tailwind ベースの既存UIと噛み合わず、1次元向けの調整余地が乏しい |

**方針**: `lib/tanaoroshi/scanner.ts` に薄いラッパを置き、`BarcodeDetector` が使えれば優先、なければ `@zxing/browser` にフォールバックする。呼び出し側はラッパのみを使う。

```ts
// 読取フォーマットを Code 39 のみに限定する（誤読低減 + 高速化）
hints.set(DecodeHintType.POSSIBLE_FORMATS, [BarcodeFormat.CODE_39]);
hints.set(DecodeHintType.TRY_HARDER, true);
```

```
package.json への追加: "@zxing/browser", "@zxing/library"
```

### 5.2 読取仕様

| 項目 | 仕様 |
|---|---|
| カメラ | `getUserMedia({ video: { facingMode: "environment", width: { ideal: 1920 }, focusMode: "continuous" } })`。**1次元は解像度が効くため 2次元前提より高めに要求する**。HTTPS 必須（Amplify は充足） |

### 5.3 読取結果の正規化と検証（誤読対策）

Code 39 にはチェックディジットが無いため、**アプリ側で書式検証を行う**。Phase 0 の実測により、極めて強力な検証ルールが使えることが判明した。

**実測結果**（`scripts/analyze-tanaoroshi-itemcode.ts` / 1,699品番）

| 項目 | 実測 |
|---|---|
| 長さ | **全1,699件が6文字固定**（例外なし） |
| 文字種 | 英大文字 + 数字のみ。3パターン `A99999`(1,558件) / `AA9999`(92件) / `AAA999`(49件) |
| Code 39 で表現不可の文字 | **0件**（全品番が Code 39 でエンコード可能） |
| 先頭文字 | H:785 / F:642 / T:257 / R:8 / S:7 |
| **他品番の接頭辞になっている品番** | **0件** |

**正規化・検証処理**（`lib/tanaoroshi/scanner.ts`）

```ts
function normalizeItemCode(raw: string): string | null {
  // ① Code 39 の start/stop 文字を除去（デコーダによって含まれる場合がある）
  let c = raw.trim().replace(/^\*+|\*+$/g, "");
  // ② 大文字化・全角→半角
  c = c.toUpperCase().replace(/[Ａ-Ｚ０-９]/g, (s) => String.fromCharCode(s.charCodeAt(0) - 0xfee0));
  // ③ 書式検証: 英数字6文字固定（Phase 0 実測。truncation 誤読をここで弾く）
  if (!/^[A-Z0-9]{6}$/.test(c)) return null;
  return c;
}
```

- **`*` の除去は必須**。ZXing は通常 start/stop を除去して返すが、実装・経路によっては `*F00001*` のまま返る。除去しないとカタログ照合が全件失敗する
- **6文字固定チェックが truncation 誤読への主防御**。部分読み取りは長さが変わるため、ここでほぼ確実に弾ける
- 接頭辞衝突が0件のため、「短く読めたコードがたまたま別の実在品番と一致する」事故は起きない
- 検証NGのときはエラー音を鳴らし「読み取れませんでした。もう一度かざしてください」と表示（**登録しない**）

## 7. 主要ロジック設計

### 7.1 差分リスト発行（回数確定）— F-07

**Lark Base にトランザクションが無いため、途中失敗しても再実行で同じ結果になる手順にする。**

```
POST /api/tanaoroshi/diff/issue { period, warehouses[] }

倉庫ごとに:
  1. TN_WH_STATUS を再読込。ステータスが「発行処理中」なら 409 で中断（二重実行防止ロック）
  2. ステータスを「発行処理中」に更新
  3. TN_ENTRY から 期ID+倉庫+現在回数+状態=有効 を取得し、品目単位に SUM
  4. TN_SYSTEM_STOCK から当該倉庫の在庫を取得
  5. 突合（倉庫＋品目単位）
     - 実棚 ≠ システム在庫 → 差分行
     - システム在庫あり & 報告なし → 実棚 0 として差分行（F-07）
     - 実棚あり & システム在庫なし → 差分行（差分数 = 実棚数量）
     - 直近入力の差分理由コードを差分行に転記
  6. 既存の同一(期,倉庫,回数)差分行を削除 → batchCreate（再実行の冪等性）
  7. 前回回数の差分行のうち、今回差分が消えた品目に 解消フラグ=true
  8. TN_WH_STATUS 更新:
       差分0件 or 現在回数 == 3 → ステータス「締め」
       それ以外                 → 現在回数 += 1、ステータス「N回目実施中」
     差分件数・報告済品目数もここで更新
  9. TN_AUDIT に「差分リスト発行」を記録
 10. Lark通知（F-10）
```

**回数の意味**: 要件どおり「差分リストを発行した時点で N 回目として確定」。発行前に入力中のデータは常に `現在回数` として記録され、発行によって確定する。

### 7.2 確定値の決定と基幹出力 — F-11

```
倉庫＋品目ごとに:
  採用値 = max(棚卸回数) を持つ回の実棚数量
         （その回で報告が無い品目は、報告のあった最大回数の値を採用）
  1回目でも報告が無い品目 → 0
```

**出力先（Phase 0 実査により確定）**

1. **`棚卸在庫情報 tbl8pCg48KRx8710` へ書き戻す**（11列）。対象期・倉庫の既存行を削除してから `batchCreateBaseRecords`（再実行安全）。業務ユーザが Lark 上で結果を確認できる
2. **EXCEL（.xlsx）でダウンロード**（基幹側が EXCEL 取込のため。業務確認済み 2026-07-21）。既存の `xlsx` パッケージを使用し、`棚卸在庫情報` と同じ11列・同じ列順・同じ列名でシートを生成する

CSV は作らない。文字コードの懸念（Shift_JIS）は EXCEL 形式により解消するため、**残課題 R-7 はクローズ**。

`lib/tanaoroshi/export.ts` に**列マッピング定義を1箇所に集約**し、テーブル書き戻しと EXCEL 生成の両方が同じ定義を参照する（二重管理しない）。レイアウト変更時はそこだけを直せば済む。

**数値パースの注意**: システム在庫情報の `在庫数` はテキスト型かつ金額列はカンマ区切り（`"75,480"`）。`lib/tanaoroshi/aggregate.ts` に `parseStockNumber()` を用意し、全参照をそこに通す。

### 7.3 未報告品目リスト — F-05

完全にクライアント側で算出する（通信不要）:

```
未報告 = catalog（対象品目） − 自端末queue内の品目 − サーバ既報告品目
```

サーバ既報告品目は `bootstrap` 時と「終了」押下時に `GET /api/tanaoroshi/entries` で取得して同期する。複数人で同一倉庫を分担する運用があるため、自端末の入力だけで判定しない。

### 7.4 取消・修正 — F-03

| 操作 | 未送信データ | 送信済データ |
|---|---|---|
| 直前取消（Undo） | queue から削除するだけ | `entries/void` で状態を「取消」に更新 + TN_AUDIT 記録 |
| 数量修正 | queue のレコードを書き換え | 旧レコードを「取消」＋新レコードを登録（`取消元実績ID` で連結） |
| 削除 | queue から削除 | 状態を「取消」に更新 |

**制約**: `TN_WH_STATUS.現在回数` が、その実績の `棚卸回数` より大きい場合（＝発行済み）は 409 を返し変更不可（F-03の制約どおり）。

### 7.5 システム在庫情報の取込・初期化 — F-13 / F-14（追加要件）

#### 前提（業務確認済み 2026-07-21）

- システム在庫情報は月次で **洗い替え**（前月分を残さない）
- 取込元は EXCEL。管理者が S-09 画面からアップロードする
- 初期化対象は **システム在庫情報**と**棚卸在庫情報**の2テーブル

#### 設計の要：クライアント側チャンク分割

既存の `app/api/upload/order-backlog/route.ts` は **1リクエスト内で全行を逐次 create/update** している。2,328行でこれをやると Amplify の 28秒タイムアウトに確実に当たる（既知の障害パターン）。棚卸では次の方式を採る。

```
[S-09 画面]
  ① EXCEL を XLSX.read でブラウザ側パース（サーバへ生ファイルを送らない）
  ② 列マッピング検証・プレビュー表示（先頭20行＋件数＋倉庫数）
  ③ ユーザ確認 → 実行
       ↓
  ④ POST /stock/purge { table: "stock", cursor }        ← 500件ずつ削除。完了までループ
  ⑤ POST /stock/import { rows: [...500件], batchId }    ← 500件ずつ登録。完了までループ
       ↓
  各リクエストは1〜2秒で完了。画面には「1,500 / 2,328 件」の進捗バーを表示
```

- **1リクエスト = 最大500件**（`batchCreateBaseRecords` / `batchDeleteBaseRecords` のチャンクサイズと一致）。2,328行なら 削除5回 + 登録5回 = 計10リクエスト
- 途中で失敗しても、そのチャンクから再開できる（`cursor` / `batchId` を画面が保持）
- サーバ側は逐次ループを一切書かない。必ず batch API を使う

#### 取込仕様

| 項目 | 内容 |
|---|---|
| 対象シート | **`data`**（サンプル実査済み。先頭シートをフォールバックとして許容） |
| 列マッピング | **EXCEL 48列 と Lark「システム在庫情報」48列が列名・列順ともに完全一致**（実査済 2026-07-21 / `scripts/analyze-tanaoroshi-xlsx.ts`）。**変換不要の恒等写像**。ヘッダー行が既知の48列と一致するかを検証し、不一致ならエラーで中止する |
| 日付列 | `締日` `繰越日` は Lark 側 DateTime。`XLSX.read(buf, { cellDates: true })` で Date として読み、epoch ms で書き込む。**`最終仕入日` は Lark 側が Text 型**（実データに全角空白が入っている）なので文字列のまま扱う |
| 数値列 | Lark 側で Number 型なのは `倉庫コード` `調整入庫数` `調整出庫数` `棚卸数` `調整数` `返品数量` `返品金額` の7列のみ。**残りは Text 型**なので文字列のまま投入し、既存データとの整合を保つ |

| 型変換 | 既存テーブルの型に合わせる。**在庫数などは Text 型のまま**（既存データと整合させる。安易に Number へ変えると既存の月次運用や基幹側と齟齬が出る） |
| 検証 | 倉庫コード・品番が空の行はエラーとして行番号付きで報告し、取込を中止（部分取込しない） |
| 重複検証 | **倉庫コード+品番の重複を検出したらエラーで中止**。突合単位の一意性が壊れると棚卸全体が破綻するため |
| 上限 | `MAX_IMPORT_SIZE`（既存 `lib/upload-validation.ts`）を流用 |

### 7.5.1 テーブル件数上限対策（F-15・業務確認済み 2026-07-22）

**背景**: 在庫棚卸はロット別管理をしていないため、同一倉庫・同一品目が複数ロット分だけ読み取られる。要件 F-06 どおり「1読取＝1レコード」で個別保持するため、**1回の棚卸で約7,000件**が登録される見込み（実測: システム在庫の倉庫×品目の組合せは2,328。平均約3ロット/品目）。1回目＋再棚卸（2・3回目）で1棚卸あたり1万件近くなり、**Lark Base のテーブル上限2万件に数ヶ月で到達**する。

**方針（業務判断）**: 個別レコード保持は維持する（監査性優先。集約はしない）。代わりに**棚卸の締め後に、その棚卸データをアーカイブ（EXCEL保存）してから初期化（削除）する運用**で上限を回避する。

**アーカイブ→初期化フロー（S-09 に機能追加。ワンクリック方式）**

```
前提: 全倉庫が締め済み かつ 基幹連携出力（棚卸在庫情報への書き戻し＋EXCEL）完了
  ① S-09 で「アーカイブして初期化」ボタンを押す → 確認ダイアログで承認
  ② 自動で 棚卸_実績・差分リストを EXCEL ダウンロード（監査保管用）
  ③ 続けて 棚卸_実績・差分リスト・倉庫進捗 を 500件チャンクで全件削除
  ④ 次の棚卸は新しい棚卸期で開始
```

- **一括・ワンクリック**（2026-07-22 UX変更）: 個別選択・テーブル名手入力は廃止。ボタン1つで「アーカイブDL→関連テーブル一括削除」を実行。ユーザー確認は `window.confirm` ダイアログ1回のみ。
- **初期化対象**: `棚卸_実績`（膨張するのはここ）＋ `差分リスト` ＋ `倉庫進捗` をまとめて削除。`棚卸_期` は履歴として残す（消さない）。
- **アーカイブ強制**: 削除の前に必ずアーカイブEXCELを自動ダウンロードする（実績・差分の2ファイル）。基幹書き戻し済みでも、読取単位の生ログ（写真・入力方式・時刻・在庫状態）は棚卸_実績にしか無いため、監査証跡として保管する。
- **サーバ保護は維持**: purge API の `confirmName`（テーブル名一致）チェックは残し、クライアントが正しい名称を自動送信する。外部から誤ってAPIを叩いても誤削除しにくい。
- **安全ガード**: 「実施中」の棚卸期があるときは棚卸_実績の初期化を拒否（作業中データの消失防止）。§7.5 の在庫初期化と同じ楽観ガード。
- **監査**: `棚卸_操作履歴` に「初期化」を件数付きで記録。

**API**: `GET /api/tanaoroshi/archive?table=entry|diff`（全件→EXCEL生成）、初期化は既存 `POST /api/tanaoroshi/stock/purge` の対象テーブルを拡張（`entry` / `diff` / `wh_status`）。

**実装状況（2026-07-22 完了）**: S-09 に「棚卸データのアーカイブと初期化」セクションを追加。**「アーカイブして初期化」ボタン1つで、確認ダイアログ→実績・差分のEXCEL自動DL→実績・差分・倉庫進捗を一括削除**（`runArchiveAndPurge`）。`PURGEABLE_TABLES` に `needsActivePeriodGuard` フラグを追加し、entry/diff/wh_status/stock は実施中期があると初期化拒否。purge の `confirmName` チェックはサーバ保護として残し、クライアントが正しい名称を自動送信。archive route は写真等の添付列をファイル名/トークン文字列に落として出力。本番ビルド確認済み。

**将来最適化（今回は不採用）**: 倉庫＋品目＋回数で1レコードに集約すれば件数は約2,328件/回に減る（読取明細はJSON列で保持）。要件 F-06 の「個別レコード保持」との兼ね合いで今回は見送り。上限運用が回らなくなった場合の第一候補として記録しておく。

### 7.6 システム在庫に無い品目の扱い（業務確認済み 2026-07-21）

要件 F-02 は「マスタ未登録はエラーで登録しない」としていたが、**業務判断により「差分として出力する」に変更**する。棚卸の目的は現物の把握であり、システムに載っていない現物こそ差分として上げるべきという判断。

| ケース | 挙動 |
|---|---|
| システム在庫にある品番 | 通常どおり品名・単位を表示して数量入力 |
| システム在庫に無いが**品目マスタにある** | 品名を表示。警告音 + 確認ダイアログを経て登録。差分リストに 理論数0 で出力 |
| どちらにも無い品番 | 品番のみで登録可。**品名は空欄**。差分リストに「品名未確定」として出力し、生産管理部が確認する |

差分リスト・基幹出力での表現:

```
理論数 = 0
数量   = 実棚数量
差異数 = 実棚数量（プラス差分）
備考   = "システム在庫なし"
```

**品目マスタ権限（R-1'）の位置づけ**: 品目マスタが読めれば品名が自動で埋まり運用が楽になるが、**読めなくても品番のみで成立する**設計にしてある。したがって R-1' は依然として v1 のブロッカーではない（Phase 2 で取得できれば品名解決を追加する）。

**依存追加**: `@zxing/browser` `@zxing/library`（package.json）

**冪等送信の実装**: entryId=クライアント採番UUID。`submitEntries` がバッチ内IDを OR フィルタで既存検索（50件ずつ）→未登録のみ batchCreate。`flushQueue` は accepted∪duplicated を queue から削除。→ 通信断・二重タップ・再起動でも二重計上しない。

**bootstrap**: 実施中の期が無ければ `period:null` とエラーメッセージ。倉庫指定時はカタログ（システム在庫ベース）＋倉庫進捗（回数）＋報告済み品目コードを1回で返す。倉庫名はクライアント（倉庫一覧）が保持するため空返し。

**Phase 1 のスコープ外（Phase 2/3 で実装）**: 取消・修正（F-03）、未報告品目リスト（F-05）、写真添付、在庫状態フラグUI（現状は良品固定）、差分理由コード、進捗ダッシュボード（S-05）、差分リスト発行（S-06）。scan画面は在庫状態=良品固定・写真なしで登録する。

**未実施の検証**: 実機でのCode39読取率（R-14）、iOS/Androidでのカメラ・音・機内モード復帰。認証必須のため自動確認不可 → ローカル`npm run dev`ログイン or デプロイ後に実機テストが必要。

**メニュー未登録**: `/tanaoroshi`（PGM054）は画面が動くようになったが `register-tanaoroshi-menu.ts` は未実行。`/seizou/tanaoroshi`（PGM055・S-05ダッシュボード）は Phase 3 まで画面が無いため、メニュー登録は Phase 3 完了後にまとめて実施予定。

## 11. テスト方針

- **単体（vitest）**: `lib/tanaoroshi/aggregate.ts`（加算・取消の集計、差分算出、確定値決定）と `export.ts` を純関数として重点的にテストする。ここが業務上の正しさの中心。
- **結合**: 冪等送信（同一バッチ2回送信で件数が増えないこと）、発行の二重実行、3回目→締めの遷移。
- **実機**: iOS Safari / Android Chrome での カメラ起動・読取速度（1秒以内）・音/バイブ・機内モード復帰。
