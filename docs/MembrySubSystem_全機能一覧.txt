0. システム概要
================================================================================

MembrySubSystem は、山口産業グループの社内業務を統合する Web システムです。
売約(受注案件)を軸に、営業・設計・工務・製造・総務・経営の各部門の業務機能を
1つの Next.js アプリケーションへ集約し、データストアには Lark Base (Bitable) を
使用しています。

■ 特徴
  - 全画面が Lark OAuth によるシングルサインオンで保護される
  - メニュー表示マスタ／機能配置マスタ／グループ権限／個別権限の4マスタで
    画面単位の表示権限を制御。サーバ側 API でも同じ権限を強制
  - 業務データは Lark Base 上にあり、業務ユーザーが Lark 側から直接閲覧・修正可能
  - AWS Amplify (SSR / WEB_COMPUTE) 上でマルチブランチ運用

■ 技術スタック
  フレームワーク  : Next.js 14 (App Router) / React 18 / TypeScript 5
  スタイル/UI     : Tailwind CSS 3, Radix UI (slot), lucide-react, cva
  データストア    : Lark Base (Bitable)  ※@larksuiteoapi/node-sdk 1.56
  認証            : Lark OAuth (OIDC) + 自前 JWT (jose) / next-auth は一部残存
  状態・取得      : @tanstack/react-query, 自前 fetchJson
  フォーム        : react-hook-form + zod
  グラフ          : Recharts 3, frappe-gantt 1.2
  PDF/画像        : @react-pdf/renderer, react-pdf, pdfjs-dist, jspdf,
                    html2canvas, react-zoom-pan-pinch, qrcode, pixelmatch
  ファイル        : xlsx, jszip, react-dropzone, date-fns
  AI              : @anthropic-ai/sdk (原価分析・売上分析・OCR)
  外部ストレージ  : Box Platform (参考図面PDF, Client Credentials Grant)
  監視            : Sentry (@sentry/nextjs) ※本番のみ有効
  デプロイ        : AWS Amplify (amplify.yml)
  テスト          : vitest
  開発補助        : Miyabi (自律エージェントフレームワーク)

     API: GET /api/customer-requests?seiban=

     当該製番の品質不具合情報一覧
     API: GET /api/quality-issues?seiban=

 [5] 作業日報       (nippou)
     現場作業日報を「受付コード→施工業者」でグルーピング表示。
     写真表示、業者別ZIP/一括ダウンロード、日報投稿用QRコード生成・mailto送付
     API: GET /api/nippou, POST /api/nippou/anken, GET /api/file, /api/file/proxy

 [6] 工事写真台帳   (photo-ledger)  ※詳細は 1-4
     API: /api/nippou, /api/koji-ledger/*, /api/baiyaku-detail, /api/file/proxy

 [7] ガントチャート (gantt-chart)
     紐づくガントチャート表示 + 社内工程表14工程(受注〜完了検査)バー、
     部署別(設計/鉄工/縫製/工務)日程バー、工程表取込
     API: /api/gantt, /api/schedule, /api/eigyo/gantt/charts,
          /charts/by-seiban, /holidays, POST /api/schedule/import, /init

 [8] 原価分析       (cost-analysis)
     売上金額・原価合計(予定/実績と差額の色分け)のサマリーカードと
     Recharts グラフ。AI原価分析コメント
     API: GET /api/cost-analysis, GET /api/ai-cost-analysis

 [9] 関連資料(案件書庫) (documents)
     部署別の書類種別ごとの資料一覧・PDFサムネイル・プレビュー・
     アップロード/削除・更新履歴・工程表PDFのOCR読取(結果確認/修正して保存)
     API: GET /api/documents, POST /api/documents/upload, /delete,
          GET|POST /api/documents/history, POST /api/ocr/schedule, GET /api/file

 [10] 資料ダウンロード (bulk-download)
     部署単位/全体の全選択・部分選択で複数資料を選び JSZip で一括ダウンロード
     API: GET /api/documents, GET /api/file/proxy

 その他: 画像差分表示(ImageDiff)、PDFサムネイル(PdfThumbnail)、QRコード生成

9. 認証・認可基盤
================================================================================

 middleware.ts
   全リクエストで auth-token Cookie の JWT(HS256/jose)を検証するグローバル
   認証ゲート。
   ・未認証時、API は 401 JSON(HTMLリダイレクトによる res.json() 破綻を回避)、
     画面は /auth/signin?callbackUrl= へ
   ・NEXTAUTH_SECRET 未設定時は fail-closed で拒否(フォールバック鍵は撤去済み)
   ・除外: api/auth, api/lark-auth, api/batch, api/genba, genba, auth,
     _next/static, _next/image, favicon.ico

 /api/lark-auth
   Lark OIDC でユーザーを認証し、自前 JWT を auth-token Cookie として
   発行/検証/削除する中核エンドポイント。GET=セッション確認、DELETE=ログアウト。
   オープンリダイレクト対策として NEXTAUTH_URL を優先し x-forwarded-host を
   信用しない。

 lib/auth.ts        クライアント用フック useAuth() / signOut()
 lib/auth-server.ts サーバー(API Route)用 getServerSession()
 lib/auth-options.ts NextAuth authOptions(Lark CredentialsProvider、JWT戦略、
                    maxAge 4時間)※一部残存

 lib/menu-permission.ts
   Lark Bitable の4マスタ(メニュー表示/機能配置/グループ権限/ユーザー権限)から
   ユーザー別の許可メニュー構造を構築する。
   ・getEmployeeByEmail / getEmployeeByLarkId で社員解決
   ・expandDepartmentChain(): Lark Contact 部門ツリー(5分キャッシュ)を親方向へ
     辿り「仙台営業所→東日本営業部→営業部」のような部門チェーンへ展開し
     グループ権限を継承
   ・buildUserPermissions / buildPermittedMenuStructure / getAllMenuStructure

 lib/menu-access.ts
   画面URL単位のメニュー権限をサーバ側でも強制するガード requireMenuAccess()。
   ・機能配置マスタを実行時に url_path → program_id 解決(60秒キャッシュ)し、
     マスタ変更へ自動追従
   ・未ログイン401 / 未許可403。マスタ未登録URLは fail-closed で管理者(PGM040)
     のみ許可
   ・読取(GET)は制限せず書込系で使う設計

 lib/kpi-permission.ts / kpi-audit.ts
   resolveKpiUser() / KPI_PROGRAMS / requireKpiProgram() による
   プログラムID単位の権限判定と監査。SEISAN_MASTER(PGM040)がマスタ管理者相当。

 lib/batch-auth.ts
   cron バッチAPI用の Bearer 認証 batchUnauthorized() と定数時間比較 safeStrEqual()。
   BATCH_SECRET 未設定/不一致は 401 の fail-closed。

 /api/menu-permission
   ログインユーザーの権限メニュー構造を返す(サイドバー構築用)。
   mode=all / mode=masters は requireKpiProgram(SEISAN_MASTER) で管理者限定。

 認証フロー
   /auth/signin → Lark OAuth 認可URL → /auth/lark-callback → JWT Cookie 発行
   → セッション(employeeId / employeeName / department / email)確立
   → /api/menu-permission で 個別権限(優先) or グループ権限 を解決
   → 許可メニュー構造をサイドバーに描画

 主な環境変数: LARK_APP_ID / LARK_APP_SECRET / LARK_BASE_TOKEN /
               LARK_BASE_TOKEN_MASTER / LARK_SYARYO_BASE_TOKEN /
               NEXTAUTH_SECRET / NEXTAUTH_URL / BATCH_SECRET / CRON_SECRET /
               ANTHROPIC_API_KEY / BOX_* / SENTRY_*
 注意        : Amplify SSR は実行時 env を読めないケースがあり、
               コード側のフォールバック定数も併用している。
               env はブランチ単位で上書きされるため、設定はブランチごとに確認。
