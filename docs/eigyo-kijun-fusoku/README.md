# 基準風速・垂直積雪量 検索機能 — 要件定義 / 設計書

営業メニューに「地域から建築基準法上の **基準風速 (V0)** と **垂直積雪量** を検索する」機能を追加する。
元になった Excel ツール `基準風速・垂直積雪量検索.xlsx`（2026年4月時点データ）を Web 化する。

> **本書の前提（確定 / 一部未確定）**
> - **データ保持方式**: ✅ **確定 = Lark Bitable master base に参照テーブルを新設し全行登録**
>   （業務ユーザーが Lark 上で直接編集・更新できることを優先）。→ [§7](#7-データ保持方式)
> - **標高依存の積雪計算**: v1 は固定値検索のみ、標高計算は v2（未確定・推奨案）。→ [§9](#9-標高依存の垂直積雪量計算)

---

## 1. 背景・目的

- カーポート・テラス・物置等の構造計算／見積において、設置地域の **基準風速** と **垂直積雪量** は
  必須パラメータ。現状は営業担当が Excel ツールで都度引いている。
- Excel はファイル配布・バージョン管理・同時利用に難があり、membrysubsystem の営業メニューへ統合して
  「ブラウザで地域を選ぶだけで両値が出る」状態にする。
- 出典は各自治体 HP／建築基準法施行令。**データは年1回程度の更新**（公開データ）。

## 2. 用語

| 用語 | 意味 |
|------|------|
| 基準風速 (V0) | 建築基準法施行令87条に基づく地域別の風速 (m/s)。30〜46程度。 |
| 垂直積雪量 | 同86条に基づく地域別の積雪深 (cm)。一部地域は標高依存。 |
| 区分1〜3 | 同一市区郡内をさらに細分する地域区分（旧地名・大字・エリア等）。段階的に絞り込む。 |
| 標高識別 | その地域の積雪量が標高に応じて変わるか否かのフラグ（元データ S 列「〇」）。 |
| 海率 | 風速算定上の海岸線比率。**本ツールは考慮しない**（元ツール仕様踏襲）。 |

| **積雪・風速元データ** | **7,661** | **中核ルックアップ表**。下記キー列＋風速・積雪・標高計算情報 |
| 区分1マスタ | 1,141 | 区分1のプルダウン候補（市郡区→区分1） |
| 区分2マスタ | 4,882 | 区分2のプルダウン候補（県名/区分→区分2） |
| 区分3マスタ | 1,798 | 区分3のプルダウン候補 |
| 風速マスタ | 1,010 | 風速の参照元（全国風速一覧。未掲載地は30m/s） |

### 3.1 中核テーブル「積雪・風速元データ」の列

| 列 | 名称 | 用途 |
|----|------|------|
| B | 県番号 | 1〜47 |
| D | 県名 | 検索キー① |
| E | 市・郡・区 | 検索キー② |
| F | 区分1 | 検索キー③（任意） |
| G | 区分2 | 検索キー④（任意） |
| H | 区分3 | 検索キー⑤（任意） |
| Q | 風速 (m/s) | **出力①**（全7,661行に値あり） |
| R | 積雪量 (cm) | **出力②**（6,080行に固定値。残り1,457行は標高計算） |
| S | 標高計算 有無 | 「〇」=標高依存 |
| T | 符号 | 標高計算の比較演算子（例 `<=`） |
| U | 基準値 | 標高のしきい値（例 570[m]） |
| W | 積雪算出方法 | 標高→積雪の算出方法（自治体により記述） |
| I〜P, V, X | 風速識別・備考等 | 補助 |

**キー充足統計**: 区分1あり 6,703 / 区分2あり 2,142 / 区分3あり 961。
→ 大半は「県＋市郡区(＋区分1)」で確定。区分2・3は一部地域のみ。

## 4. 機能要件

### FR-1 地域カスケード検索
- 入力: 県名 → 市・郡・区 → 区分1 → 区分2 → 区分3 を**プルダウンで段階選択**。
- 上位を選ぶと下位の選択肢が動的に絞り込まれる（区分が無い地域は以降のプルダウンを非活性）。
- 一意に確定した時点で結果（基準風速・垂直積雪量）を表示。

### FR-2 結果表示
- **基準風速 (m/s)** と **垂直積雪量 (cm)** を大きく表示。
- 出典・備考（元データ V/W 列）と「※海率は考慮していません」を併記。
- データ基準時点（例「2026年4月時点」）を明示。

### FR-3 標高入力（標高依存地域）※ v2
- 標高識別「〇」の地域は、固定積雪値の代わりに **標高(m) 入力欄**を表示。
- 入力標高と元データの符号(T)・基準値(U)・算出方法(W)から垂直積雪量を算出。
- v1 では当該地域に「この地域は標高計算が必要です（v2対応予定）」と明示し、固定値地域のみ算定。

### FR-4 検索補助
- 市・郡・区名のインクリメンタル検索（部分一致での絞り込み）。
- 「該当なし／未掲載地」の場合、風速はデフォルト30m/sの注記を表示（元ツール仕様）。

### FR-5 結果の利用
- 表示値のコピー（クリップボード）。
- （任意・将来）見積/構造計算機能への引き渡し。

## 5. 非機能要件

- **性能**: 検索操作は体感即時（< 200ms）。→ 初回に Lark から全件取得しツリー化、以降はクライアント側絞り込み＋APIキャッシュで担保（§10）。
- **認証/権限**: 既存 middleware のセッション必須配下。営業メニュー権限を持つユーザーが利用可。
- **保守性**: 元 Excel 更新時、変換スクリプト1本でデータ再生成できること（§8）。
- **可搬性**: Next.js App Router / AWS Amplify SSR / 既存規約（`"use client"`+`force-dynamic`）に準拠。

## 7. データ保持方式

**採用: Lark Bitable master base に参照テーブルを新設し全行（7,661行）登録**（他マスタと同じく全環境共有）。

理由:
- 風速・積雪は法改正や自治体更新で変わるため、**業務ユーザーが Lark 上で直接編集・更新できる**ことを優先。
  再デプロイ不要で値を直せる（社員マスタ・メニューマスタ等と同じ運用）。
- 既存規約（マスタは Lark Bitable・項目名は日本語）に統一できる。

### 7.1 新規テーブル「基準風速・積雪量マスタ」（master base）

項目名は日本語（業務ユーザーが直接見るため）。書込/参照フィールド:

| フィールド名 | 型 | 元Excel列 | 備考 |
|--------------|----|-----------|------|
| 県名 | テキスト | D | 検索キー① |
| 市・郡・区 | テキスト | E | 検索キー② |
| 区分1 | テキスト | F | 検索キー③（任意） |
| 区分2 | テキスト | G | 検索キー④（任意） |
| 区分3 | テキスト | H | 検索キー⑤（任意） |
| 基準風速 | 数値 | Q | m/s |
| 垂直積雪量 | 数値 | R | cm（標高依存時は空） |
| 標高計算有無 | チェック/単一選択 | S（〇判定） | 標高依存フラグ |
| 標高符号 | テキスト | T | `<=` 等（v2） |
| 基準標高 | 数値 | U | m（v2） |
| 積雪算出方法 | テキスト | W | v2 |
| 備考 | テキスト | V | |

- 配置先 base: **master base**（`getLarkBaseTokenForMaster()`）。`lib/lark-tables.ts` の
  `TABLE_BASE_CONFIG` に新テーブルを **master 指定**で追加し、テーブルID定数を定義する。
- 突合キー: `県名+市・郡・区+区分1+区分2+区分3` の複合（インポートの冪等性判定に使用）。

## 8. データ投入（Excel → Lark）

- スクリプト: `scripts/kijun-fusoku/import-to-lark.ts`（`xlsx` で読取 → `batchCreateBaseRecords`/
  `batchUpdateBaseRecords` で投入。本調査で xlsx 読取の導通確認済み）。
- 入力: `基準風速・垂直積雪量検索.xlsx`「積雪・風速元データ」シート（ヘッダ R7、データ R8〜）。
- 列マッピング: D=県名, E=市・郡・区, F=区分1, G=区分2, H=区分3, Q=基準風速, R=垂直積雪量,
  S=標高計算有無(〇→true), T=標高符号, U=基準標高, W=積雪算出方法, V=備考。
- **バッチ集約**: 500件/リクエストで `batchCreate`（逐次ループ禁止＝Amplify28秒タイムアウト/空ボディ対策の既知教訓）。
- **冪等**: 既存レコードを複合キーで突合し、新規=create / 差分=update / 一致=skip。再実行で差分0に収束させる。
- **空上書き防止**: Excel側が空の項目で既存値を潰さない（社員マスタ同期で得た教訓を踏襲）。
- バリデーション: 基準風速が数値・空でない／県名が47種に収まる／複合キー重複を警告。
- 既定 dry-run（差分サマリのみ）/ `--execute` で実投入（書込先は本番 master base のため明示フラグ）。
- 更新運用: 業務ユーザーは通常 **Lark 上で直接編集**。新Excel一括差替え時のみ本スクリプトを再実行。

## 9. 標高依存の垂直積雪量計算（v2）

- 対象 1,457 行（S列「〇」）。固定値の代わりに **標高(m) を入力して積雪量を算出**する。
- 元データの符号(T)・基準値(U)・算出方法(W)で決まるが、**W列の算出式の正規化に追加解析が必要**
  （サンプル上 W が空で T/U のみの行が多く、しきい値判定型と式型が混在）。
- v2 着手時タスク:
  1. S=〇 の全1,457行について T/U/W のパターンを分類（しきい値型／係数式型／参照元参照型）。
  2. 「積雪参照元」シート（自治体別 垂直積雪量）との突合で算出規則を確定。
  3. 算出関数を `lib/kijun-fusoku-snow.ts` に実装し、UIに標高入力欄を追加。
- v1 では当該地域に「標高計算が必要（v2対応予定）」と明示し、固定値地域（6,080行）で先行リリース。

## 10. API 設計

Lark Bitable 採用のため**サーバAPIが必須**（クライアントから直接 Lark は叩かない）。

- `app/api/eigyo/kijun-fusoku/route.ts`（GET）
  - `export const dynamic = "force-dynamic"`。
  - `getBaseRecords(テーブルID, { baseToken: getLarkBaseTokenForMaster(), pageSize: 500, pageToken })`
    を `has_more` が無くなるまでループして全7,661行を取得。
  - 取得結果を `{県名→市郡区→区分…}` のツリー（プルダウン候補）と検索用配列に整形して返す。
- **性能対策（重要）**: 7,661行は pageSize 500 で約16ページ。毎リクエストの全件読込は数秒かかるため、
  **モジュールスコープのインメモリキャッシュ（TTL 数分〜数時間）**で温かいコンテナ内は再利用する。
  - Amplify SSR はサーバレスでコールド時はキャッシュ非永続のため、初回/コールドは数秒許容（社内ツール）。
  - 代替最適化（必要時）: 整形済みツリーJSONを定期バッチで生成して配信、または県単位で分割取得。
- クライアント fetch は `lib/fetch-json.ts` の `fetchJson` を使用（空ボディ/HTML/504を明確なエラーに）。
- カスケード絞り込みは初回ロードしたツリーをクライアント側で展開（県選択→市候補→区分候補…）。

## 13. テスト観点

- 投入: Excelデータ行数 = Lark登録レコード件数。風速空欄ゼロ。県名47種。再実行で差分0（冪等）。
- 検索: 代表地域（例 北海道/白老郡/白老町=風速34/積雪70）の期待値一致。区分2/3を要する地域の絞り込み。
- 境界: 未掲載地のデフォルト風速注記。標高依存地域の注記表示（v1）。
- 非機能: 初回ロード後の検索操作が体感即時。APIキャッシュ有効時の応答時間。
