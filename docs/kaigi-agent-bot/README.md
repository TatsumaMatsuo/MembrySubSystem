## 1. 背景と目的

### 1.1 背景
- 社内会議はLarkのチャットルーム単位で運営されており、議事録はLarkのAI要約（妙記／AI Notes）で自動生成されている。
- しかし「要約が生成されるだけ」で止まっており、**次のアクションに繋がっていない**。
- 次回会議の議題設定・宿題の追跡・資料準備はすべて事務局の手作業に依存し、属人化している。

### 1.2 目的
議事録AI要約を入力として、以下を**チャット上の会話だけで完結**させる。

1. 会議内容の構造化（決定事項／宿題／未決事項／リスク）
2. 次回会議のアジェンダ案・確認事項・検討事項の自動生成
3. 定型文プロンプトによる各種資料（議事録清書・アジェンダ表・報告資料）の自動作成
4. 宿題（ToDo）の担当者・期限管理とリマインド

### 1.3 基本方針
- 既存の社内AIエージェント **Menkuma（メンくま）** の基盤（Lark Bot／LLM／RAG／HITL承認フロー）を流用し、新規構築を最小化する。
- **人を置き換えず、事務局の作業を増幅する**（デジタル武装）。最終確定は必ず人が承認する。

## 2. 現状課題（As-Is）

| No | 課題 | 影響 |
|---|---|---|
| C-01 | AI要約は生成されるが読まれず放置される | 決定事項が共有されない |
| C-02 | 次回議題を事務局が毎回ゼロから起票 | 準備に毎回30〜60分 |
| C-03 | 宿題（持ち帰り事項）の追跡が個人任せ | 期限超過・積み残しの常態化 |
| C-04 | 前回未決事項が次回に引き継がれない | 同じ議論の再発 |
| C-05 | 会議資料の体裁・粒度が担当者ごとにバラバラ | 読み手の負担、意思決定の遅延 |
| C-06 | 過去の会議での決定事項を検索できない | 「前に決めたはず」が探せない |

## 3. スコープ

### 3.1 対象範囲（v1.0）
- Larkチャットルームに常駐するBot（1グループ＝1会議体を想定）
- 対象会議体：定例会議（部門定例、DX推進会議、営業部会議 等）※初期は2〜3グループでPoC
- 議事録AI要約テキストの取得・解析
- アジェンダ／確認事項／検討事項の生成
- 定型プロンプトによる資料生成（Lark Docs / Lark Base 出力）

### 3.2 対象外（v1.0では実装しない）
- 音声のリアルタイム文字起こし（Lark妙記の既存機能を利用）
- 会議中のリアルタイム進行制御（タイムキープ発話等）→ v2.0以降
- 社外者を含む会議への適用
- 人事評価・機密度の高い会議体への適用

## 4. 利用シナリオ（To-Be）

### シナリオA：会議終了直後
1. Lark妙記がAI要約を生成し、チャットルームに投稿される。
2. Botが投稿を検知（または事務局が `/議事録取込` を送信）。
3. Botが要約を構造化し、**決定事項／宿題／未決事項**に分解してチャットに返信。
4. 事務局がチャット上で修正指示（例：「宿題3の担当を営業部に変更」）。
5. 確定内容をLark Baseに登録。

### シナリオB：次回会議の3営業日前
1. Botが自動起動し、前回未決事項＋期限到来の宿題＋新規上申事項を集約。
2. **次回アジェンダ案**を生成しチャットに提示（議題／目的区分／担当／想定時間）。
3. 事務局が `/確定` を送信 → Lark Docsにアジェンダを出力、カレンダー招待に添付。

### シナリオC：資料が欲しいとき
1. 事務局が `/資料作成 経営報告` のように定型プロンプトを送信。
2. Botが該当期間の議事録データを参照して報告資料のドラフトをLark Docsに生成。
3. URLをチャットに返信。人が加筆・確定。

## 5. 機能要件

### 5.1 機能一覧

| ID | 機能名 | 概要 | 優先度 |
|---|---|---|---|
| F-01 | 議事録取得 | チャットルーム内のAI要約・妙記データを取得 | 必須 |
| F-02 | 議事録構造化 | 決定事項／宿題／未決／リスクへLLMで分解 | 必須 |
| F-03 | 次回アジェンダ生成 | 未決＋宿題＋新規議題からアジェンダ案を生成 | 必須 |
| F-04 | 確認事項・検討事項抽出 | 「確認が必要」「要検討」の論点を抽出・列挙 | 必須 |
| F-05 | 定型プロンプト応答 | 定型コマンドを解釈し該当処理を実行 | 必須 |
| F-06 | 資料自動生成 | Lark Docs／Base／スライド形式で出力 | 必須 |
| F-07 | タスク管理・リマインド | 宿題の担当・期限を管理し自動督促 | 高 |
| F-08 | 会議前後の自動通知 | 開催3日前アジェンダ／終了後要約の自動配信 | 高 |
| F-09 | 過去議事録横断検索 | RAGで過去の決定事項を検索・回答 | 中 |
| F-10 | HITL承認 | 生成物は人の承認を経て確定・配信 | 必須 |
| F-11 | 監査ログ | 入出力・操作者・生成物を記録 | 必須 |

### 5.2 F-01 議事録取得
- 取得方式（いずれかを採用。PoCで検証）
  - (a) Bot常駐チャットの新規メッセージをイベント購読し、妙記／AI要約の投稿を検知
  - (b) 事務局が要約テキストを`@Bot`宛に貼り付け
  - (c) Lark妙記APIから会議単位で要約・文字起こしを取得 ※**API提供範囲は要確認**
- 取得対象：会議名、開催日時、参加者、AI要約本文、（可能なら）文字起こし全文
- 添付ファイル（Docs／Base／PDF）が投稿されている場合はリンクを併せて保持

### 5.3 F-02 議事録構造化
LLMで以下のJSONスキーマに正規化する。

```json
{
  "meeting_id": "MTG-20260724-001",
  "meeting_name": "DX推進定例会",
  "date": "2026-07-24",
  "attendees": ["松尾", "…"],
  "decisions": [
    { "no": 1, "content": "決定内容", "owner": "担当", "decided_at": "2026-07-24" }
  ],
  "actions": [
    { "no": 1, "content": "宿題内容", "owner": "担当", "due": "2026-08-05", "status": "未着手" }
  ],
  "pending": [
    { "no": 1, "content": "未決事項", "reason": "情報不足", "next_step": "次回に見積提示" }
  ],
  "confirmations": [
    { "no": 1, "content": "確認事項", "to": "確認先", "deadline": "2026-07-31" }
  ],
  "considerations": [
    { "no": 1, "topic": "検討事項", "background": "背景", "options": ["案A", "案B"] }
  ],
  "risks": [
    { "no": 1, "content": "リスク・懸念", "level": "中" }
  ]
}
```

**制約**
- 議事録に記載のない内容をBotが創作しないこと（根拠なき推測の禁止）。
- 担当者・期限が不明な場合は空欄とし、`要確認` フラグを立ててチャットで人に問い返す。

### 5.4 F-03 次回アジェンダ生成
- 入力：前回の `pending` ＋ 期限到来／超過の `actions` ＋ 事務局が追加した新規議題
- 出力項目：議題No／議題名／**目的区分（報告・協議・決議）**／担当／想定時間／関連資料リンク／起因（前回未決 等）
- 合計想定時間が会議設定時間を超過する場合は警告し、優先度による絞り込み案を提示する。

### 5.5 F-04 確認事項・検討事項の抽出
- 発言中の未確定表現（「〜だと思う」「確認します」「後で詰める」等）を検出し、確認事項候補として抽出。
- 複数案が並立したまま結論が出ていない論点を検討事項候補として抽出し、判断に必要な情報（誰に何を確認すべきか）を併記する。

### 5.6 F-05 定型プロンプト仕様（チャットコマンド）

`@Bot` メンション＋定型文で起動する。表記ゆれ吸収のため自然文も許容する。

| コマンド | 動作 | 出力先 |
|---|---|---|
| `/議事録取込` | 直近のAI要約を取得し構造化して提示 | チャット |
| `/決定事項` | 当該会議体の決定事項一覧を出力 | チャット／Docs |
| `/宿題一覧` | 未完了の宿題を担当・期限付きで一覧化 | チャット |
| `/次回アジェンダ` | 次回会議のアジェンダ案を生成 | チャット→Docs |
| `/確認事項` | 確認事項・検討事項を整理して提示 | チャット |
| `/資料作成 <種別>` | 指定種別の資料ドラフトを生成 | Docs |
| `/会議準備` | アジェンダ＋前回要約＋関連資料をまとめて配信 | チャット |
| `/検索 <キーワード>` | 過去議事録を横断検索して回答（根拠リンク付） | チャット |
| `/確定` | 直前の生成物を確定しBase登録・配信 | Base／Docs |
| `/取消` | 直前の生成物を破棄 | ― |
| `/ヘルプ` | コマンド一覧を表示 | チャット |

- `<種別>` の初期セット：`議事録`／`アジェンダ`／`経営報告`／`進捗報告`／`課題一覧`
- コマンド／テンプレートは**設定ファイルまたはLark Baseで後から追加できる仕様**とする（ハードコーディング禁止）。

### 5.7 F-06 資料自動生成
- 出力形式：Lark Docs（既定）、Lark Base（レコード登録）、Lark Sheets（一覧表）
- 会社標準の様式テンプレート（見出し構成・用語）を定義し、テンプレートに沿って生成する。
- 生成物には「**AI生成ドラフト・要確認**」の注記を必ず付与する。

### 5.8 F-07 タスク管理・リマインド
- 宿題はLark Baseの「会議タスク」テーブルで一元管理。
- 期限3日前／当日／超過時に担当者へDM通知。
- 会議冒頭に前回宿題の進捗一覧をチャットへ自動投稿。

### 5.9 F-10 HITL承認
- Botの生成物は**すべてドラフト扱い**。事務局の `/確定` 操作を経て初めてBase登録・関係者配信を行う。
- 確定操作者と確定日時をログに残す。

## 6. 非機能要件

| 分類 | 要件 |
|---|---|
| 応答性能 | 定型コマンドの一次応答 5秒以内、生成完了 60秒以内（処理中は「作成中」を表示） |
| 可用性 | 平日8:00–20:00を対象。障害時は手作業運用へ縮退可能とする |
| セキュリティ | 議事録データは社内ネットワーク／自社AWS環境内で処理。個人情報・機密語は送信前にマスキング |
| 権限管理 | Botが参加するチャットルームのメンバーのみ利用可。`/確定` は事務局ロールに限定 |
| 監査 | 入力プロンプト・出力・実行者・対象会議IDを全件ログ保存（保存期間1年） |
| 保守性 | プロンプト・テンプレート・コマンド定義を外部設定化し、非エンジニアでも改訂可能とする |
| 拡張性 | 会議体の追加はチャットルームへのBot追加＋設定登録のみで完了すること |

## 7. システム構成（案）

```
Lark チャットルーム
   │  ①メンション／定型コマンド、②妙記AI要約の投稿
   ▼
Lark Open Platform（イベント購読 / Webhook）
   ▼
会議進行エージェント（AWS EC2 / Node.js / pm2）
   ├─ コマンドルーター（定型プロンプト解釈）
   ├─ 議事録パーサ（LLM 構造化）
   ├─ アジェンダ生成エンジン
   ├─ 資料ジェネレータ（Docsテンプレート適用）
   ├─ RAG検索（Qdrant：過去議事録ベクトル）
   └─ ジョブスケジューラ（会議前後の自動実行）
   ▼
Lark API（IM返信 / Docs作成 / Base登録 / Calendar参照）
```

- LLMは既存Menkuma基盤の構成（社内LLM＋必要に応じ外部LLM）を踏襲。機密度に応じルート分離。
- ベクトルDB（Qdrant）に過去議事録を蓄積し、F-09の横断検索に利用。
- **【精査 P3】** イベント購読の常駐Bot（long-running）のため EC2/pm2 を採用。ただし**運用主体・監視・シークレット管理（Secrets Manager）・月次コスト**を I-07 で確定すること。既存の社内AI基盤（Menkuma／#32）と**同一Botに相乗りするか、別Botとして常駐させるか**は I-06 の結論に従う（相乗りなら本節の常駐サーバは既存基盤へ集約）。

## 8. データ設計（Lark Base）

| テーブル | 主なフィールド |
|---|---|
| 会議体マスタ | 会議体ID、会議体名、チャットID、開催サイクル、事務局、資料テンプレート |
| 会議録 | 会議ID、会議体ID、開催日、参加者、要約本文、原文リンク、取込日時 |
| 決定事項 | 決定ID、会議ID、内容、担当、決定日 |
| 会議タスク | タスクID、会議ID、内容、担当、期限、状態、完了日 |
| 未決・検討事項 | 事項ID、会議ID、内容、区分（確認／検討）、次アクション、繰越回数 |
| アジェンダ | アジェンダID、会議体ID、開催予定日、議題No、議題名、目的区分、担当、想定時間、確定フラグ |
| 実行ログ | ログID、日時、実行者、コマンド、対象会議ID、結果 |

- 「繰越回数」を持たせ、**3回以上繰り越された事項は警告表示**（塩漬け防止）。
- **【精査 P7】** 会議体マスタに「事務局（確定権限者）」を保持し、`/確定` の権限判定に用いる。
- **【精査 P5/I-08】** RAG横断検索（F-09）は**会議体単位のアクセス制御**を前提とし、利用者が参加していない会議体の議事録は検索・回答の対象に含めない。

## 9. 外部インタフェース／事前確認事項

| No | 確認事項 | 備考 |
|---|---|---|
| I-01 | Lark妙記（AI要約）の外部API取得可否・取得範囲 | 不可の場合はチャット投稿テキストの取得で代替 |
| I-02 | Botのチャット常駐・メンション受信の権限スコープ | Lark管理者にてアプリ権限申請 |
| I-03 | Lark Docs／Base／Calendar のAPI利用権限 | 同上 |
| I-04 | 会議録データの保存場所・保存期間に関する社内規程 | 情報システム／総務と調整 |
| I-05 | 議事録を外部LLMへ送信する場合の可否基準 | 原則社内LLM。機密区分の定義が必要（精査 P4） |
| **I-06** | **既存 Menkuma／社内AI案件（#32・`shainai-help-bot`）との関係。新規常駐Botか既存Bot拡張か。事務局ロール（確定権限）の管理方法** | 構築範囲・運用主体に直結（精査 P1/P7）。Phase 0 で確定 |
| **I-07** | **常駐サーバ（AWS EC2/pm2＋Qdrant）の運用主体・監視・シークレット管理・月次コスト目安** | DX推進室運用を想定（精査 P3） |
| **I-08** | **RAG横断検索（F-09）のアクセス制御単位（利用者が参加する会議体のみ検索対象とするか）** | 権限逸脱防止（精査 P5） |

※ I-01〜I-03はLark Open Platformの最新仕様確認が必要。実装着手前にPoCで疎通検証すること。
※ **I-01（妙記AI要約の取得可否）は本システム全体の前提**。Phase 0 の着手ゲート条件とし、取得不可時は手貼り運用（F-01(b)）へ縮退する前提で握ること（精査 P2）。

## 12. リスクと対策

| リスク | 対策 |
|---|---|
| AI要約の精度不足により構造化が誤る | HITL承認を必須化。原文リンクを常に併記し検証可能にする |
| Botが議事録にない内容を生成（ハルシネーション） | 出典（発言箇所）の引用を必須化。根拠なき項目は`要確認`表示 |
| 機密情報の外部流出 | 社内LLM優先、マスキング処理、対象会議体の限定 |
| 事務局がBotを使わず形骸化 | 定型コマンドを5つ以内に絞り、会議前自動配信で「使わなくても効果が出る」設計にする |
| 会議体ごとの様式差異 | テンプレートを会議体マスタで切替可能とする |
