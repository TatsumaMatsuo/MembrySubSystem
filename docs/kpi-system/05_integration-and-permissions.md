# 05. 既存システム統合・6ロール権限設計

本システムを既存 MembrySubSystem の**メニュー権限フレームワーク**に正しく載せるための設計。

## 1. 既存メニュー権限システムの理解(前提)

`lib/menu-permission.ts` / マスタ用 Base の4テーブルで構成:

| テーブル(env) | 役割 |
|----------------|------|
| メニュー表示マスタ(`LARK_TABLE_MENU_DISPLAY`) | 3階層メニュー(L1トップ/L2カテゴリ/L3機能)。`メニューID`/`親メニューID`/`階層レベル`/`表示順` |
| 機能配置マスタ(`LARK_TABLE_FUNCTION_PLACEMENT`) | プログラム(画面)を `配置メニューID` に紐付け。`URLパス` を持つ |
| グループ権限マスタ(`LARK_TABLE_GROUP_PERMISSION`) | **部署名**単位の許可。`グループ名`/`対象種別(menu/program)`/`対象ID`/`許可フラグ` |
| 個別権限マスタ(`LARK_TABLE_USER_PERMISSION`) | 社員ID単位の許可(個別があれば**個別優先**、グループ無視) |

**権限解決ロジック**(`buildUserPermissions`):
1. ログイン → `getServerSession()` → email/open_id で社員特定(`getEmployeeByEmail` / `getEmployeeByLarkId`)。
2. 社員の部署を Lark Contact ツリーで親階層展開(`expandDepartmentChain`。例: 仙台営業所→東日本営業部→営業部)。
3. 個別権限があればそれを、なければ展開した部署名でグループ権限を適用。
4. 明示的に許可された menu/program のみ表示(`buildPermittedMenuStructure`)。

> **重要**: 既存は「**表示/編集/非表示**」の粗い権限(`PERMISSION_LEVELS`)を menu/program 単位で持つ。KPIシステムが要求する「**自部門の施策だけ編集可**」のような**データ行レベルの権限**は既存に無い → 拡張が必要(後述 §3)。

## 2. メニュー登録(マスタ投入)

新規トップメニュー `生産本部KPI` を3階層で登録する。`scripts/` 配下に既存の `register-syaryo-menu.ts` / `register-syaryo-admin-pgm.ts` と同様の投入スクリプトを用意(本フェーズでは設計のみ)。

### メニュー構造(案)

```
L1: M0XX 経営【新設・独立メニュー】
 └ L2: M0XX-01 経営計画
     ├ program: 中期経営計画ダッシュボード /keiei/dashboard
     ├ program: 全社KPI(年度計画 vs 実績) /keiei/company-kpi
     ├ program: 会計データ入力(月/四半期/半期/年) /keiei/kaikei-input
     └ program: 中計マスタ管理(管理者)  /keiei/midterm

L1: M0YY 生産本部KPI
 └ L2: M0YY-01 KPIレビュー
     ├ program: ダッシュボード      /seisan-kpi/dashboard
     ├ program: KPI実績入力         /seisan-kpi/input
     ├ program: 施策管理(PDCA)     /seisan-kpi/measures
     ├ program: ★達成評価          /seisan-kpi/stars
     └ program: 過去実績参照        /seisan-kpi/history
 └ L2: M0YY-02 KPI管理(管理者)
     ├ program: KPIマスタ管理       /seisan-kpi/master
     ├ program: グループマスタ管理  /seisan-kpi/master(タブ)
     └ program: データエクスポート  /seisan-kpi/export
 └ L2: M0YY-03 共通
     └ program: ヘルプ(運用ガイド)  /seisan-kpi/help
```

- `MENU_GROUPS`(`lib/lark-tables.ts`)に `"経営"` と `"生産本部KPI"` を追加。
- L1/L2/program の `メニューID` は既存採番規則(`M001-02-01` 形式)に合わせる(実 ID は投入時に確定)。
- **ヘルプ**は全画面ヘッダの「？」からも開く(メニュー登録は閲覧導線として併設)。

## 3. 6ロールの権限マッピング

要件 §3 の6ロールを、既存「部署ベースのグループ権限 + 個別権限」に**メニュー表示権限**としてマップしつつ、不足する**データ行レベル制御**を新設する。

### 3.1 メニュー表示レベル(既存フレームで実現)

| ロール | 表示する program | 実現方法 |
|--------|-----------------|---------|
| 経営層 | 中計DB / 全社KPI / 生産本部DB / ★達成 / 過去実績(閲覧) | グループ権限(役員部署)に該当 program を許可 |
| 本部長 | 全 program(経営+生産本部)。会計データ入力を含む | 個別権限 or 生産本部長の部署グループに全許可 |
| 部長・次長 | 生産本部DB/入力/施策/★/過去/ヘルプ | 部署グループ(製造部・生産管理部)に許可 |
| 課長 | 生産本部DB/施策/★/過去/ヘルプ(+入力?) | 部署グループ(各課)に許可 ※スコープ確認 |
| 閲覧者 | ダッシュボード/★/過去/ヘルプ | 部署グループに閲覧 program のみ許可 |
| 管理者 | 全 program + マスタ管理 + グループ管理 + 中計マスタ | DX推進室グループ or 個別権限 |

> **経営メニュー**(中計・全社KPI・会計入力)は経営層・本部長・管理者中心。**会計データ入力は本部長**(+管理者)に限定。ヘルプは全ロール閲覧可。

### 3.2 データ行レベル(新設が必要)

「**自部門のデータだけ編集できる**」を実現するため、API 側で**ロール × 担当部署**による行フィルタ/書込ガードを追加する。

**実装方針(案)**:
- 新テーブル `SEISAN_KPI_ROLE`(または既存個別権限に種別拡張)で **社員 → KPIロール + 担当部署スコープ** を定義。
  | フィールド | 例 |
  |-----------|----|
  | 社員ID/メール | … |
  | KPIロール | 本部長/部長/課長/閲覧/管理者 |
  | 担当部署スコープ | `製造部`(部長) / `本社鉄工課`(課長) / `*`(本部長・管理者) |
- API(`/api/seisan-kpi/*`・`/api/keiei/*`)は session→社員→KPIロールを解決し、
  - **読取**: 経営/本部長/管理者/閲覧=全件、部長=自部+配下課、課長=自課。グループ別表示は所属部署スコープに従う。
  - **書込**:
    - 実績入力(`actuals`)=担当(生産管理課)+管理者
    - 施策の計画/実施/効果(`pdca`)=担当グループの責任者のみ / 本部長コメント・翌月アクション=本部長のみ
    - **会計データ(`kaikei-actual`)=本部長**(+管理者)
    - KPIマスタ/期切替/**グループマスタ**/**中計マスタ**=管理者
- **担当部署スコープは Lark部署で解決**: 既存 `getEmployeeByEmail` + `expandDepartmentChain`(Lark Contact 部門ツリー)で社員の部署→配下を判定。部署マスタ=Lark部署と一貫。
- サーバ側で必ず検証(クライアント非表示だけに頼らない=セキュリティ要件)。

> 簡易版: ロール定義を最初は**部署名から推定**(部署=製造部かつ役職=部長 → 部長ロール)し、明示テーブルは段階導入でもよい。ただし役職情報の取得元(社員マスタに役職列があるか)を要確認。

## 4. 既存データとの関係

| 既存 | KPIシステム | 方針 |
|------|------------|------|
| `COMPANY_KPI`(全社KPI) | 経営KPI(売上67億/ROA/労働生産性) | 全社値は `COMPANY_KPI` を**参照**。本システムは生産本部スコープに限定し重複入力を避ける(ダッシュボードで参照表示するか要確認) |
| `SALES_KPI`(営業部KPI) | — | 別部門。連携不要(将来 全社KPIダッシュボード統合の余地) |
| `EMPLOYEES`(社員マスタ) | ロール解決の入力 | 既存 `getEmployeeByEmail` を流用。役職列の有無を確認 |
| メニュー権限4テーブル | メニュー表示 | そのまま流用(マスタ投入のみ) |

## 5. 認証・セッション

- 既存 `getServerSession()`(`lib/auth-server.ts`)+ Lark OAuth(`app/api/auth/*`)をそのまま使用。新規認証は不要。
- 各 page は既存同様、サーバ側でセッション確認 or クライアントから権限API呼び出し。

## ❓要確認(統合・権限分)

1. **役職情報の所在**: 社員マスタ(`EMPLOYEES`)に役職(本部長/部長/課長)列があるか。無ければ KPIロールは明示テーブル必須。
2. **課長ロールのスコープ**(再掲): 入力権限を持つか(要件は「施策入力」。実績入力は生産管理課担当)。
3. **経営KPIの重複**: 全社経営値を `COMPANY_KPI` 参照にするか、本システムで独立入力するか(現状 Excel は本システム側で持つ)。
4. **メニューID採番**: 新トップメニューの L1 ID を既存体系のどこに割り当てるか(採番ルール確認)。
5. **個別権限 vs グループ権限の運用**: KPIロールは個別権限ベース推奨か(部署一括だと役職差を表現しづらい)。
